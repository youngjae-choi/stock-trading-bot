# Agent Onboarding

이 문서는 이 저장소에서 작업하는 모든 AI Agent와 VS Code Assistant의 단일 온보딩 기준 문서다.
작업 전 이 문서를 읽고, 이후 문서 읽기 순서와 우선순위는 `DOC_HIERARCHY.md`를 단일 기준으로 따른다.

## 1. 목적

- 여러 도구가 같은 작업 규칙으로 움직이도록 한다
- 같은 파일을 서로 다른 방식으로 수정해 충돌하는 일을 줄인다
- PM 승인 흐름, 구현 순서, 검증 기준을 한 번에 이해하도록 한다

## 2. 기본 원칙

- PM은 방향과 우선순위를 결정한다
- Sisyphus는 메인 지휘자 역할을 담당한다. **OpenCode 와 Claude Code(Opus 컨트롤러)를 공동 주력**으로 두며, 한 세션의 메인 지휘자는 하나로 유지한다
- 구현/탐색/리뷰는 **각 환경의 실행 주체**가 담당한다 (OpenCode: Codex CLI/Gemini CLI 등 / Claude Code: **Task 서브에이전트**)
- 역할·프로세스·게이트는 **environment-agnostic** 이며, 환경별 매핑은 `DOC_HIERARCHY.md` 4절을 따른다
- 같은 파일을 여러 도구가 동시에 수정하지 않는다
- 승인되지 않은 작업 범위 확장은 금지한다
- 추측 구현은 금지한다
- 완료 보고는 실제 검증 이후에만 한다

## 3. 세션 시작 순서

`DOC_HIERARCHY.md` 2항의 읽기 순서를 따른다.
읽기 순서의 단일 기준은 `DOC_HIERARCHY.md`에만 정의하며, 이 문서에서 별도로 순서를 재정의하지 않는다.

## 3-1. 요청 문서 선확인 규칙

새 작업은 아래 문서를 먼저 확인하고 누락 항목을 보완한다.

- `FEATURE_TEMPLATE.md`: 요청/완료 기준/테스트 항목 체크
- `UI_BASELINE.md`: UI 상태/레이아웃/접근성 기준
- `ERROR_HANDLING.md`: 오류 분류/메시지/로그/복구 기준

요청서가 짧거나 모호하면 바로 구현하지 말고 `FEATURE_TEMPLATE.md` 형식으로 재정리 후 진행한다.

## 4. 표준 작업 흐름

모든 개발/수정은 아래 흐름을 기준으로 진행한다.

1. 개발/수정 요청 발생
2. **새 작업 시작 전 커밋 여부 확인** → Sisyphus가 PM에게 반드시 질문: "현재 상태를 커밋(저장)하시겠습니까?"
3. 기획서 업데이트
4. 작업계획서 + 테스트계획서 작성 및 PM 승인
5. 구현 수행
6. 코드 리뷰 + 브라우저 확인
7. 테스트결과서 작성
8. 매뉴얼 + 기획서 + WBS 업데이트

중요:
- **Step 2 커밋 확인 없이 새 작업 착수 금지** — 커밋은 PM이 결정한다
- **main 통합(merge·push) 권한은 메인 지휘자만 가진다.** 커밋 방식은 환경별로 다르다 — OpenCode는 Sisyphus만 커밋(Codex/Gemini는 git commit 금지), Claude Code 서브에이전트 구동은 서브에이전트가 기능 브랜치에 원자 커밋 후 지휘자가 merge
- Step 4 승인 전 구현 금지
- Step 6 검증 없이 완료 선언 금지
- 기존 기능 영향 범위 파악 전 수정 금지

## 5. 역할별 책임

### Sisyphus / OpenCode
- 요구사항 해석
- 전문 Agent 배정과 작업 범위 통제
- PM 승인 게이트 관리
- 기술 결정 이유 설명
- 최종 변경 설명 정리
- git commit 수행

Sisyphus는 PM의 단일 창구이지만 모든 일을 직접 하지 않는다. 코드 탐색, 공식 문서 확인, 계획서 작성, 구현, 리뷰, UI 작업은 가능한 한 전문 Agent에게 위임한다.

### Prometheus / OpenCode
- 계획서, 테스트계획서, 테스트결과서, 매뉴얼/WBS 정리
- PM이 이해할 수 있는 언어로 변경 내용 문서화

### Metis / OpenCode
- 모호한 요청의 숨은 의도, 누락된 요구사항, 실패 가능성 분석
- 구현 전 질문이 필요한 지점 정리

### Explore / OpenCode
- 코드베이스 구조, 활성 코드 경로, 기존 구현 패턴 조사
- 수정 후보 파일과 영향 범위 요약

### Librarian / OpenCode
- 공식 문서, API 스펙, 외부 라이브러리 사용법 확인
- 추측 구현 방지를 위한 근거 자료 제공

### Executor / Codex CLI
- 승인된 계획 범위 안에서 코드 구현
- 통합 시점 안정성 검토
- 로컬 테스트 및 정밀 수정
- 함수/컴포넌트 주석 및 로그 작성
- 기존 기능 파괴 위험 확인
- Sisyphus가 작업 범위와 완료 조건을 명확히 지정해 호출

### Frontend / Gemini CLI
- React 컴포넌트 작성
- UI 작업
- 프론트엔드 상호작용 구현

### Oracle / Codex CLI
- 아키텍처, 보안, 성능, 회귀 위험 검토
- 테스트 결과와 완료 주장 검증

### Momus / OpenCode
- 작업계획서와 테스트계획서의 누락/모호함/검증 불가능 항목 검토

### Multimodal Looker / OpenCode
- 스크린샷, PDF, 다이어그램, 이미지 기반 오류 상태 해석

## 6. 파일 수정 규칙

- 같은 파일을 두 도구가 동시에 수정하지 않는다
- 구현 전 수정 대상 파일 목록을 먼저 정한다
- 대형 변경은 한 도구가 초안을 만들고 다른 도구가 통합/리뷰한다
- 문서 수정과 코드 수정을 한 세션에서 섞을 때는 변경 목적을 명확히 분리한다
- 기존 사용자의 변경사항은 절대 임의로 되돌리지 않는다

추천 분담 (OpenCode 환경 기준. Claude Code 환경에서는 괄호의 실행 주체로 매핑):
- PM 대화/승인/커밋: Sisyphus (Claude Code: 세션 본체=Opus 컨트롤러)
- 문서/계획서/결과서: Prometheus (Claude Code: 지휘자 직접 또는 Plan 서브에이전트)
- 코드 구조 조사: Explore (Claude Code: Explore 서브에이전트 또는 직접 Grep/Glob)
- 공식 문서/API 조사: Librarian (Claude Code: WebFetch/context7 또는 서브에이전트)
- 비프론트엔드 구현: Executor / Codex CLI (Claude Code: Task 서브에이전트, 기계적=sonnet)
- 프론트엔드 UI 구현: Frontend / Gemini CLI (Claude Code: Task 서브에이전트)
- 보안/성능/회귀 검토: Oracle (Claude Code: code review 서브에이전트 / `/code-review`)
- 소규모 수정(1~2곳)은 지휘자가 직접 처리 가능

## 7. 구현 규칙

- TypeScript strict mode 유지
- 기술 스택 임의 변경 금지
- 새 라이브러리 추가 전 PM 승인 필요
- 모든 함수/컴포넌트에 목적과 파라미터를 설명하는 주석 작성
- 복잡한 비즈니스 로직은 단계별 주석 작성
- 주요 작업은 시작/완료/실패 로그를 남긴다
- 에러 처리 없이 기능 구현하지 않는다
- API 키, 토큰, 비밀번호를 코드나 로그에 노출하지 않는다

## 8. 검증 규칙

완료 선언 전 아래를 확인한다.

- 실제 활성 코드 경로를 기준으로 수정했는가
- 공식 문서 확인이 필요한 부분을 검증했는가
- 테스트계획서의 시나리오를 점검했는가
- 서버 로그에서 정상 흐름을 확인했는가
- 브라우저에서 주요 동작을 확인했는가

### ⚠️ 테스트 완료 기준 (PM 지시 — 모든 Agent/Claude 필수 준수)

**개발 완료 = 코드 작성 완료가 아니다. 아래 테스트까지 통과해야 완료다.**

1. **API 호출 테스트**: 변경된 기능과 연관된 API endpoint를 직접 호출하여 응답 검증
2. **E2E 테스트**: 전체 통과 확인 (새 기능 추가 시 시나리오 1개 이상 필수 추가)
3. **빌드 검증**: 빌드 에러 0개 확인

### ⚠️ 에러 발생 시 완료 보고 금지 규칙 (PM 지시 — 필수 준수)

- **에러가 존재하는 상태에서 완료 보고를 하지 않는다**
- 에러 발생 시 → 원인 파악 → 수정 → 재확인 → 에러 없음 확인 후 보고
- "에러가 있지만 완료됐습니다" 보고는 허용되지 않는다
- 혼자 해결 불가능한 에러는 "이 에러는 PM이 직접 해결해야 합니다: [이유]" 형식으로 명확히 구분해서 보고한다

## 9. PM 보고 규칙

- 계획 승인 전에는 구현하지 않는다
- 에러 보고는 `로그 확인 → 원인 설명 → 해결책 제안` 순서로 한다
- 완료 후에는 다음 추천 작업 3가지를 제안한다
- 불확실한 내용은 추측하지 말고 `확인 필요`로 표시한다

## 9-1. 요청 수신 시 선행 검토 규칙 (필수)

PM의 요청을 받으면 **즉시 실행하기 전에** 아래 두 가지를 먼저 검토한다.

1. **왜(Why)**: PM이 이 방향을 요청한 이유가 무엇인가? 근본 문제는 무엇인가?
2. **대안(Alternative)**: 요청한 방법보다 더 빠르거나, 더 안전하거나, 더 근본적인 해결책은 없는가?

검토 결과:
- 더 나은 대안이 있으면 → 먼저 제안하고 PM 선택을 받은 뒤 진행한다
- 요청한 방향이 최선이면 → 이유를 한 줄로 확인 후 진행한다
- "요청했으니 바로 실행"은 금지한다

## 10. 새 세션용 체크리스트

작업 시작 전 아래 체크리스트를 빠르게 확인한다.

- 나는 지금 어떤 역할인가
- 지금 작업은 계획 단계인가 구현 단계인가 검증 단계인가
- PM 승인이 필요한 상태인가
- 수정할 파일이 다른 도구와 겹치지 않는가
- 문서 확인이 필요한 외부 API가 있는가
- 완료 후 어떤 검증까지 해야 하는가

## 11. 우선 참고 문서

전체 문서 목록과 우선순위는 `DOC_HIERARCHY.md`를 참조한다.

## 12. Agent Mailbox (에이전트 간 통신)

- 위치: `docs/agent-comm/INBOX_[페르소나].md`, `OUTBOX_[페르소나].md`
- 용도: PM이 복잡한 프롬프트를 직접 입력하지 않고 파일로 전달
- 파일명은 AI 도구가 아닌 **페르소나 이름** 기준

## 13. 한 줄 운영 요약

메인 지휘자(Sisyphus 역할)가 계획·게이트·작업 위임을 담당한다. 지휘 환경은 **OpenCode 와 Claude Code 공동 주력**이며, 구현은 각 환경의 실행 주체(OpenCode: Codex/Gemini CLI / Claude Code: Task 서브에이전트)가 맡는다. 모든 작업은 PM 승인과 검증 기준을 먼저 따른다.
