# ERROR_HANDLING.md

## 목적

이 문서는 기능 구현 시 발생할 수 있는 오류와 예외 상황에 대해  
**사용자 경험**, **UI 동작**, **로그 기록**, **복구 방법**을 일관되게 처리하기 위한 기준을 정의한다.

목표는 다음과 같다.

- 오류를 "예상 밖의 사고"가 아니라 "기본 설계 대상"으로 다룬다
- 사용자에게는 이해 가능한 안내를 제공한다
- 개발자에게는 추적 가능한 로그를 남긴다
- AI 에이전트가 정상 흐름만 구현하고 끝내지 않도록 한다
- 장애, 실패, 빈 응답, 지연, 부분 성공 등 현실적인 문제를 공통 규약으로 다룬다

이 문서는 개별 기능 명세보다 우선하지 않는다.  
다만 개별 기능 명세에 별도 지시가 없으면 본 문서 기준을 기본값으로 따른다.

연계 문서:
- UI/상태 표현 기본 규칙은 `UI_BASELINE.md`를 따른다.
- 구현/테스트 게이트는 `DOC_HIERARCHY.md`의 계층 기준을 따른다.

---

## 핵심 원칙

### 1. 오류는 예외가 아니라 기본 흐름의 일부다

기능은 정상 케이스만 구현해서는 안 된다.

모든 기능은 최소한 아래를 기본 고려한다.

- 잘못된 입력
- 필수값 누락
- 네트워크 실패
- 요청 지연
- 서버 오류
- 권한 부족
- 데이터 없음
- 중복 요청
- 부분 성공
- 사용자 취소
- 외부 API 불안정
- 예기치 않은 응답 형식

---

### 2. 사용자와 개발자에게 보여주는 정보는 다르다

사용자는 문제를 해결하거나 다음 행동을 결정할 수 있어야 한다.  
개발자는 원인을 추적할 수 있어야 한다.

따라서 오류 처리 시 다음을 분리한다.

- 사용자 메시지
- 개발자 로그
- 모니터링/추적 정보

금지:
- stack trace를 사용자에게 그대로 노출
- 내부 클래스명/함수명/raw error 메시지 그대로 노출
- "알 수 없는 오류"만 반복
- 아무 로그도 남기지 않고 삼킴

---

### 3. 오류가 나도 가능한 한 사용자 상태를 보존한다

실패했다고 해서 사용자가 방금 입력한 값이나 작업 맥락이 사라지면 안 된다.

원칙:
- 폼 제출 실패 시 입력값 유지
- 검색 실패 시 기존 결과를 무조건 지우지 않음
- 저장 실패 시 사용자가 다시 시도할 수 있게 유지
- 편집 실패 시 이전 상태와 변경 상태를 구분 가능하게 유지

---

### 4. 재시도 가능한 실패와 불가능한 실패를 구분한다

모든 실패를 같은 방식으로 처리하면 안 된다.

구분 예:
- 재시도 가능: 네트워크 일시 오류, timeout, 일시적 서버 문제
- 사용자 수정 필요: 입력 형식 오류, 필수값 누락, 권한 부족
- 관리자 대응 필요: 설정 누락, 내부 서버 오류 반복, 외부 연동 키 문제

---

### 5. 조용히 실패하지 않는다

요청 실패, 저장 실패, 삭제 실패, 부분 반영 실패는 반드시 사용자 또는 로그 관점에서 감지 가능해야 한다.

금지:
- 버튼 클릭 후 아무 반응 없음
- 실패했는데 성공처럼 보이는 UI
- 일부만 성공했는데 전체 성공처럼 표시
- 로그 없이 UI만 실패

---

## 오류 분류 기준

기능 구현 시 오류를 다음 범주로 나누어 처리한다.

### 1. Validation Error
사용자 입력이 요구 조건을 만족하지 못하는 경우

예:
- 필수값 누락
- 이메일 형식 오류
- 숫자 범위 초과
- 문자열 길이 초과
- 중복된 이름 입력

처리 원칙:
- 가능한 한 필드 가까이 표시
- 어떤 값을 어떻게 고쳐야 하는지 알려준다
- 제출 전 검증과 제출 시 검증을 함께 고려한다

예시 메시지:
- 이름을 입력해주세요
- 이메일 형식이 올바르지 않습니다
- 비밀번호는 8자 이상이어야 합니다

---

### 2. Network Error
클라이언트가 서버에 정상적으로 도달하지 못한 경우

예:
- 인터넷 연결 끊김
- DNS 문제
- 요청 timeout
- 브라우저 네트워크 오류

처리 원칙:
- 서버 로직 문제와 구분한다
- 사용자에게 재시도 가능성을 알려준다
- 입력값 또는 현재 작업 상태는 최대한 유지한다

예시 메시지:
- 네트워크 연결이 불안정합니다. 다시 시도해주세요
- 요청 시간이 초과되었습니다. 잠시 후 다시 시도해주세요

---

### 3. Server Error
서버 내부 문제로 요청 처리에 실패한 경우

예:
- 500 Internal Server Error
- 예기치 않은 예외
- 데이터베이스 오류
- 내부 서비스 장애

처리 원칙:
- 사용자는 잘못하지 않았음을 전제로 안내한다
- 재시도 가능 여부를 상황에 따라 표시한다
- 내부 원인은 로그로 남긴다

예시 메시지:
- 요청을 처리하는 중 문제가 발생했습니다. 잠시 후 다시 시도해주세요
- 현재 서버 상태가 불안정합니다. 잠시 후 다시 시도해주세요

---

### 4. Auth / Permission Error
인증 또는 권한 부족 문제

예:
- 로그인 만료
- 접근 권한 없음
- 관리자만 가능한 기능 접근
- 조직 범위 밖 자원 접근

처리 원칙:
- 단순 실패와 구분한다
- 필요한 다음 행동을 안내한다
- 보안상 과도한 내부 정보는 노출하지 않는다

예시 메시지:
- 로그인 세션이 만료되었습니다. 다시 로그인해주세요
- 이 작업을 수행할 권한이 없습니다

---

### 5. Not Found / Empty Error
대상이 존재하지 않거나 결과가 없는 경우

예:
- 삭제된 항목 접근
- 검색 결과 없음
- 필터 결과 없음
- 상세 페이지 대상 없음

처리 원칙:
- 에러와 empty를 구분한다
- 비어 있음은 실패가 아니라 정상 상태일 수 있다
- 사용자가 다음 행동을 선택할 수 있어야 한다

예시 메시지:
- 검색 결과가 없습니다
- 요청한 항목을 찾을 수 없습니다
- 이미 삭제되었거나 접근할 수 없는 항목입니다

---

### 6. Conflict / State Error
현재 상태와 사용자 행동이 충돌하는 경우

예:
- 이미 처리된 항목 재처리
- 다른 사용자가 먼저 수정
- 최신 상태와 충돌
- 중복 저장

처리 원칙:
- 사용자의 행동이 왜 반영되지 않았는지 설명한다
- 새로고침 또는 재동기화 유도 필요 여부를 검토한다

예시 메시지:
- 이미 변경된 데이터입니다. 최신 상태를 확인해주세요
- 다른 작업에서 먼저 수정되었습니다. 다시 불러온 후 시도해주세요

---

### 7. External Dependency Error
외부 API, 결제, 메일, 파일 저장소 등 외부 시스템 문제

예:
- 결제 API 장애
- 메일 발송 실패
- 외부 검색 API 실패
- 서드파티 응답 형식 변경

처리 원칙:
- 내부 기능 실패와 구분해 로그 남김
- 부분 성공 여부를 분명히 함
- 재시도 전략 또는 대체 흐름 검토

예시 메시지:
- 외부 서비스 연동 중 문제가 발생했습니다. 잠시 후 다시 시도해주세요

---

## 사용자 메시지 작성 규칙

사용자 메시지는 아래 원칙을 따른다.

### 좋은 메시지의 조건

- 무엇이 문제인지 알 수 있어야 한다
- 사용자가 다음에 무엇을 하면 되는지 알 수 있어야 한다
- 지나치게 기술적이지 않아야 한다
- 너무 추상적이지 않아야 한다

### 권장 형식

- 문제 설명 + 다음 행동 제안

예:
- 저장 중 문제가 발생했습니다. 다시 시도해주세요
- 필수 항목을 입력한 후 다시 저장해주세요
- 네트워크 연결이 불안정합니다. 잠시 후 다시 시도해주세요

### 금지 형식

- 에러 발생
- 실패했습니다
- 알 수 없는 오류
- Exception: request failed
- InternalServerError at SaveUserAction

---

## 개발자 로그 작성 규칙

로그는 문제 추적과 재현을 위해 충분해야 한다.  
다만 민감정보는 직접 남기지 않는다.

최소 권장 항목:
- 기능명
- 액션명
- 단계
- 에러 종류
- 상태 코드
- 사용자 액션 맥락
- 요청 식별자
- 재시도 여부
- 외부 의존성 정보
- 발생 시각

예:
- feature=user-create
- action=submit-form
- stage=api-request
- error_type=network-timeout
- status_code=504
- request_id=...
- retry=false

금지:
- 비밀번호 raw 저장
- 토큰 raw 저장
- 카드번호/주민번호/민감정보 raw 저장
- 의미 없는 console.log 남발
- catch 후 무시

---

## UI 오류 처리 규약

### 1. 폼 제출 실패

기본 원칙:
- 입력값 유지
- 필드 오류는 필드 근처 표시
- 전역 오류는 상단 또는 제출 버튼 근처 표시
- 제출 버튼은 실패 후 다시 시도 가능 상태로 복구

반드시 검토:
- 어느 필드가 잘못되었는지 표시되는가
- 서버 오류와 입력 오류를 구분하는가
- 포커스를 첫 오류 필드로 이동할 필요가 있는가

---

### 2. 목록 로딩 실패

기본 원칙:
- 전체 빈 화면 방치 금지
- 실패 상태 UI 제공
- 재시도 버튼 검토
- 이전 데이터가 있다면 무조건 지우지 말고 stale view 유지 여부 검토

예시:
- 목록을 불러오지 못했습니다. 다시 시도해주세요

---

### 3. 상세 화면 로딩 실패

기본 원칙:
- 대상 없음 / 권한 없음 / 서버 실패를 가능하면 구분
- 홈으로 가기, 목록으로 돌아가기 등 복귀 행동 제공

---

### 4. 저장/수정 실패

기본 원칙:
- 사용자가 수정한 내용은 유지
- 무엇이 저장되지 않았는지 분명히 함
- 재시도 가능해야 함

---

### 5. 삭제 실패

기본 원칙:
- 삭제가 되지 않았음을 분명히 함
- 이미 삭제됐는지, 권한이 없는지, 일시 실패인지 구분 가능하면 구분
- UI에서 항목을 성급히 제거한 경우 롤백 필요

---

### 6. 부분 성공

부분 성공은 반드시 별도로 처리한다.

예:
- 저장은 성공했지만 알림 발송은 실패
- 항목 생성은 성공했지만 이미지 업로드는 실패
- 기본 데이터는 로드됐지만 부가 데이터는 실패

처리 원칙:
- 전체 성공으로 뭉뚱그리지 않음
- 성공한 것과 실패한 것을 구분해 안내
- 후속 행동 제안 필요

예시 메시지:
- 기본 정보는 저장되었지만 이미지 업로드는 실패했습니다. 다시 시도해주세요

---

## 재시도 규약

### 사용자가 직접 재시도 가능한 경우

예:
- 목록 다시 불러오기
- 저장 다시 시도
- 업로드 다시 시도

원칙:
- 버튼 또는 명확한 액션 제공
- 중복 재시도 폭주 방지
- 재시도 중 상태 표시

### 자동 재시도 가능한 경우

예:
- 일시적 네트워크 오류
- 짧은 polling 실패
- 외부 API의 간헐적 오류

원칙:
- 무한 재시도 금지
- 재시도 횟수 제한
- 사용자에게 숨길지 보여줄지 결정
- 실패 누적 시 명시적으로 알림

---

## 롤백 규약

낙관적 업데이트나 선반영 UI를 사용한 경우 실패 시 롤백 전략이 있어야 한다.

예:
- 좋아요 버튼 선반영 후 실패 시 원복
- 목록에서 삭제 선반영 후 실패 시 항목 복원
- 상태 변경 선반영 후 실패 시 이전 상태로 복구

원칙:
- 롤백이 눈에 띄게 혼란스럽지 않아야 한다
- 실패 원인을 함께 안내한다
- 조용히 원복만 하고 끝내지 않는다

---

## Timeout 처리 규약

timeout은 일반 실패와 분리해 다룬다.

원칙:
- 사용자에게 기다림이 길어졌음을 알린다
- 재시도 가능성 제공
- 같은 요청이 실제 서버에서 계속 처리 중일 수 있는지 검토한다
- 중복 요청으로 인한 부작용 가능성을 고려한다

예시 메시지:
- 요청 시간이 초과되었습니다. 처리 상태를 확인한 후 다시 시도해주세요

---

## 오프라인 / 연결 불안정 처리

앱 또는 웹이 연결 불안정 상태일 수 있음을 고려한다.

검토 항목:
- 오프라인 상태 감지 여부
- 임시 저장 필요 여부
- 재연결 후 재시도 가능 여부
- 작성 중 데이터 보존 여부

예시 메시지:
- 현재 네트워크에 연결되어 있지 않습니다
- 연결이 복구되면 다시 시도해주세요

## 권한 및 보안 관련 오류 처리

권한 오류는 단순 실패처럼 보이면 안 된다.

원칙:
- 필요한 권한이 무엇인지 과도하게 노출하지 않는다
- 로그인 필요 / 권한 부족 / 접근 불가를 구분한다
- 민감한 데이터 존재 여부를 불필요하게 암시하지 않는다

예:
- 이 항목에 접근할 수 없습니다
- 로그인 후 다시 시도해주세요

---

## API 응답 처리 규칙

AI 에이전트는 API 응답을 신뢰하지 말고 방어적으로 처리한다.

반드시 검토:
- 응답 body가 비어 있는 경우
- expected field 누락
- status는 200인데 payload가 이상한 경우
- 에러 형식이 일관되지 않은 경우
- null/undefined 처리
- 배열이 비어 있는 경우

금지:
- 응답 구조를 무조건 믿고 바로 렌더링
- 실패 응답인데 성공 흐름 진입
- null 체크 없이 접근

---

## 에러 화면 우선순위 규칙

여러 상태가 동시에 가능한 경우 우선순위를 명확히 한다.

일반 권장 순서:
1. 치명적 오류
2. 권한/인증 문제
3. 데이터 없음
4. 일반 콘텐츠
5. 보조 경고

예:
- 인증이 끊겼으면 empty state보다 로그인 유도 우선
- 치명적 로딩 실패면 부분 콘텐츠보다 에러 UI 우선

---

## 기본 금지사항

다음은 특별한 이유가 없는 한 금지한다.

- try/catch 후 아무 처리 없이 무시
- raw error를 사용자에게 그대로 노출
- 실패했는데 성공 토스트 표시
- 입력값을 날려버리는 제출 실패 처리
- 부분 성공을 전체 성공으로 처리
- 재시도 없는 일시적 오류 처리
- 무한 재시도
- 민감정보 로그 출력
- 에러 유형 구분 없이 전부 "알 수 없는 오류" 처리
- API 실패인데 UI를 정상 상태로 유지해 사용자가 오해하게 만드는 것

## 구현 체크리스트

기능 구현 시 최소 아래를 확인한다.

### 공통
- 어떤 오류 유형이 가능한가
- 사용자 메시지는 준비되어 있는가
- 개발자 로그는 남는가
- 민감정보가 노출되지 않는가
- 재시도 가능 여부가 정의되어 있는가

### 폼
- 필드 오류와 전역 오류를 구분하는가
- 제출 실패 시 값이 유지되는가
- 중복 제출 방지가 되는가

### 목록/상세
- 로딩 실패 UI가 있는가
- empty와 error를 구분하는가
- 다시 시도할 수 있는가

### 비동기 작업
- timeout 처리 기준이 있는가
- 낙관적 업데이트 실패 시 롤백하는가
- 부분 성공을 분리해 보여주는가

## AI 에이전트 작업 지침

AI 에이전트는 오류 처리 구현 시 다음을 따른다.

1. 정상 흐름만 구현하고 완료로 판단하지 않는다
2. validation, network, server, auth, empty, conflict, external dependency를 기본 검토한다
3. 사용자 메시지와 개발자 로그를 분리한다
4. 실패 시 사용자 입력과 작업 맥락을 가능한 유지한다
5. 부분 성공과 전체 성공을 구분한다
6. 재시도 가능 여부와 롤백 필요 여부를 함께 검토한다
7. 구현 보고 시 어떤 오류를 어떻게 처리했는지 명시한다

---

## 구현 보고 형식

오류 처리 관련 구현 완료 후 보고에는 최소 아래를 포함한다.

- 처리한 오류 유형
- 사용자 메시지 처리 방식
- 개발자 로그 처리 방식
- 재시도 가능 여부
- 롤백 적용 여부
- 부분 성공 처리 여부
- 테스트 또는 수동 검증 방법

## 프로젝트 특화 오류 규약 (WebRTC 헬프데스크)

아래 항목은 본 프로젝트에서 반드시 구분해 처리한다.

### 1) 미디어 권한/장치 오류
- 예:
  - 카메라 권한 거부 (`NotAllowedError`)
  - 카메라 장치 없음 (`NotFoundError`)
  - 장치 점유/실패 (`NotReadableError`, `OverconstrainedError`)

  - "카메라 권한이 필요합니다. 브라우저 설정에서 허용 후 다시 시도해주세요."
  - "사용 가능한 카메라를 찾을 수 없습니다. 장치를 확인해주세요."
- 로그 키:
  - `feature=media`, `action=getUserMedia`, `error_name`, `device_mode`, `session_id`
