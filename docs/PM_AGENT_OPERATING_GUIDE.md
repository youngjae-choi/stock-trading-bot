# PM Agent Operating Guide

이 문서는 비개발자 PM이 Sisyphus에게만 요청하고, Sisyphus가 내부 Agent를 배정해 프로젝트를 운영하는 방법을 설명한다.

## 1. PM이 할 일

PM은 기술 도구나 Agent 이름을 몰라도 된다. 아래처럼 원하는 결과를 말하면 된다.

- "이 기능이 필요해."
- "이 화면이 이상해."
- "이 오류 원인을 봐줘."
- "다음에 무엇을 해야 할지 정리해줘."
- "이 작업 커밋해줘."

Sisyphus가 요청을 기능, 버그, UI, 문서, 검증, 배포 준비 중 하나로 분류하고 필요한 Agent를 호출한다.

## 2. Sisyphus가 할 일

Sisyphus는 PM의 단일 창구다.

1. 요청의 의도와 목표를 쉬운 말로 재정리한다.
2. 누락된 요구사항, 더 나은 대안, 기존 기능 파괴 위험을 확인한다.
3. 작업 전 현재 상태를 커밋할지 PM에게 확인한다.
4. Prometheus에게 작업계획서와 테스트계획서를 작성하게 한다.
5. PM 승인 후 적절한 구현 Agent에게 작업을 위임한다.
6. Oracle 리뷰와 테스트 결과를 확인한다.
7. PM에게 결과, 검증 내용, 남은 리스크, 다음 추천 작업을 보고한다.

## 3. Agent 배정 기준

| 요청 유형 | 배정 Agent | 이유 |
|-----------|------------|------|
| 요구사항 정리, 계획서, 테스트계획서 | Prometheus | PM 언어를 실행 가능한 계획으로 바꾼다. |
| 모호한 요청, 범위가 큰 요청, 숨은 의도 확인 | Metis | 구현 전에 의도, 누락, 실패 가능성을 먼저 찾는다. |
| 코드 위치, 기존 구조, 영향 범위 조사 | Explore | 구현 전 실제 코드 경로를 확인한다. |
| 공식 문서, API 스펙, 외부 라이브러리 확인 | Librarian | 추측 구현을 막는다. |
| 백엔드, 데이터 처리, 비프론트엔드 구현 | Executor / Hephaestus | 승인된 범위 안에서 코드를 수정한다. |
| 화면, UI, CSS, 브라우저 상호작용 | Frontend | 시각적 일관성과 접근성을 맡는다. |
| 보안, 성능, 아키텍처, 회귀 위험 검토 | Oracle | 위험한 변경을 검토한다. |
| 계획서 품질 검토, 빠진 테스트 확인 | Momus | 계획이 실행 가능하고 검증 가능한지 비판적으로 본다. |
| 스크린샷, PDF, 다이어그램 해석 | Multimodal Looker | 이미지 기반 정보를 읽는다. |

## 3-1. Sisyphus 직접 수행 / 위임 기준

Sisyphus는 PM-facing 지휘자다. 아래 기준으로 직접 처리와 위임을 나눈다.

| 작업 종류 | Sisyphus 직접 수행 | 전문 Agent 우선 위임 |
|-----------|-------------------|----------------------|
| PM 요청 접수 | 요청을 쉬운 말로 재정리 | 없음 |
| 의도 파악 | 간단한 요청은 직접 정리 | 모호하거나 큰 요청은 Metis |
| 코드 구조 파악 | 이미 아는 파일 1~2개 확인 | 여러 파일/모듈 영향은 Explore |
| 외부 문서 확인 | PM에게 결과 요약 | 공식 문서/API 확인은 Librarian |
| 작업계획/테스트계획 | PM 승인 게이트 관리 | 초안 작성은 Prometheus |
| 백엔드/비즈니스 로직 구현 | 원칙적으로 직접 구현하지 않음 | Executor / Hephaestus |
| 프론트엔드/UI 구현 | 원칙적으로 직접 구현하지 않음 | Frontend |
| 아키텍처/보안/성능 판단 | 결과를 PM 언어로 설명 | Oracle |
| 테스트 작성/실패 분석 | 실행 결과를 요약 | Executor 또는 Oracle |
| 브라우저/스크린샷 확인 | 최종 확인 결과 요약 | Frontend 또는 Multimodal Looker |
| 문서/보고서 | 짧은 운영 메모 | 정식 계획서/결과서/매뉴얼은 Prometheus |
| 커밋 | Sisyphus만 수행 | 다른 Agent는 git commit 금지 |

기본값은 **전문 Agent 활용**이다. Sisyphus가 직접 처리하는 경우는 단순 질문, 커밋, 짧은 요약, 이미 충분히 확인된 단일 파일의 작은 문서 정리로 제한한다.

## 3-2. 표준 위임 흐름

새 기능, 수정, 버그 대응은 아래 흐름을 기본으로 한다.

1. Sisyphus가 PM 요청을 접수하고 의도/누락/리스크를 정리한다.
2. 범위가 불명확하면 Metis 또는 Explore에 먼저 확인시킨다.
3. Prometheus가 작업계획서와 테스트계획서를 작성한다.
4. PM 승인 후 Executor 또는 Frontend가 구현한다.
5. Oracle이 보안, 성능, 기존 기능 파괴 위험을 검토한다.
6. Sisyphus가 검증 결과와 다음 추천 작업을 PM에게 보고한다.

긴급 장애처럼 즉시 대응이 필요한 경우에도, Sisyphus는 원인 파악과 복구를 먼저 진행한 뒤 사후에 계획/검증/문서 정리를 남긴다.

## 4. 모델/비용 운영 기준

Sisyphus는 비용과 품질을 함께 관리한다.

| 역할 | 권장 모델 | 비용 기준 | 사용 이유 |
|------|-----------|-----------|-----------|
| Sisyphus | `openai/gpt-5.5` | 유료 | PM 요청 해석과 최종 판단에는 고성능 모델이 필요하다. |
| Prometheus | `openai/gpt-5.4-mini` | 유료, 상대적으로 절감 | 문서화는 품질을 유지하면서 비용을 낮출 수 있다. |
| Metis | `openai/gpt-5.4-mini` | 유료, 상대적으로 절감 | 모호한 요청의 숨은 의도와 실패 지점을 찾는다. |
| Explore | `google/gemini-3-flash-preview` | 무료/쿼터 가능 | 코드 탐색은 빠른 저비용 모델이 적합하다. |
| Librarian | `google/gemini-3-flash-preview` | 무료/쿼터 가능 | 공식 문서 탐색은 저비용 모델로 충분한 경우가 많다. |
| Executor / Hephaestus | `openai/gpt-5.5` | 유료 | 코드 수정은 오류 비용이 커서 고성능 모델을 우선한다. |
| Frontend | `google/gemini-3-pro-preview` | 무료/쿼터 또는 유료 가능 | UI 구현은 Gemini Pro 계열이 적합하다. |
| Oracle | `openai/gpt-5.5` | 유료 | 보안, 성능, 회귀 검토는 고성능 판단이 필요하다. |
| Momus | `openai/gpt-5.4-mini` | 유료, 상대적으로 절감 | 계획과 테스트 기준의 빈틈을 검토한다. |

무료/저비용 모델의 쿼터가 막히면 Sisyphus는 PM에게 어떤 Agent에 어떤 모델을 대신 배정할지, 비용 영향은 무엇인지 설명하고 선택을 요청한다.

## 5. PM에게 요청이 필요한 경우

Sisyphus는 가능한 작업은 스스로 진행한다. 다만 아래 경우에는 PM에게 확인을 요청한다.

- 현재 미커밋 변경을 저장할지 결정해야 할 때
- 새 유료 모델, 유료 API, 유료 서버 자원이 필요할 때
- OpenCode, Codex, Gemini 로그인이 풀려서 PM 계정 인증이 필요할 때
- 요구사항 선택지가 여러 개이고 결과가 크게 달라질 때
- 보안 키, 거래 API, 실제 매매와 연결되는 위험한 작업일 때

## 6. PM 요청 예시

좋은 요청은 길 필요가 없다.

```text
리뷰 화면에서 오늘 매매 결과가 잘 안 보여. PM이 보기 쉽게 개선해줘.
```

```text
이 에러 원인 찾아줘. 가능하면 고쳐줘. 기존 기능 깨지면 안 돼.
```

```text
다음 배포 전에 꼭 확인해야 할 것 정리해줘.
```

요청이 짧으면 Sisyphus가 `FEATURE_TEMPLATE.md` 기준으로 부족한 내용을 보완 질문하거나 작업계획서에 명시한다.

## 7. 완료 보고 형식

Sisyphus의 완료 보고는 짧고 명확해야 한다.

- 바뀐 내용
- 실제 검증한 내용
- 남은 리스크 또는 확인 필요 사항
- 다음 추천 작업 3가지

테스트하지 않은 내용은 완료로 말하지 않고 `확인 필요`로 표시한다.
