# 레짐별 일일예산 균등배분 포지션 사이징 — 설계서 v1.0 (확정)

작성일: 2026-06-03 · 대상: S7 주문 사이징 / S6 프리플라이트

## 원본 요구사항 (PM)

> 과감하게 사용하도록 변경하면 안되나? 예를들면 두번째 매수종목은 현재 예수금에
> 15%를 매수하는게 아니라 오늘 장개시전 예수금에 15%를 매수하도록 어때?

검토 결과 PM이 **레짐별 일일예산 ÷ 슬롯 균등배분** 방식을 채택(2026-06-03).

## 현재 동작 (변경 전)

`order_executor._calc_qty`: `qty = floor(ord_psbl_cash × position_size_pct / 100 / price)`.
- `ord_psbl_cash` = **현재 주문가능현금**(남은 현금). 30초 잔고 캐시.
- 효과: 매 주문이 *남은 현금*의 N%를 먹어 **기하급수적으로 감소** → 후순위 종목 금액이 쪼그라들고,
  Risk On이라도 활용률이 ~86%에서 천장. 자금이 과감하게 쓰이지 않음.

## 왜 바꾸나 (WHY)

- 후순위 우량 종목(TSI 높은)에 충분한 금액이 안 들어감 → 기회 손실.
- 시장이 좋아도(Risk On) 예수금을 다 활용 못함.
- PM 의도: **각 슬롯에 균등하고 의미있는 금액**을 넣어 자금을 과감히 투입.

## 왜 PM 원안(시작예수금 고정 15%)을 그대로 안 쓰나

`15% × max_positions`가 100%를 초과한다: Risk On 12×15%=**180%**, 중립 7×15%=**105%**.
→ 고포지션 레짐에서 현금 소진 시 후순위 주문이 KIS '현금부족'으로 거부되어 **체결이 선착순 운(運)**이 됨.
해결: 종목당 비율을 고정하지 말고 **레짐별 예산을 슬롯 수로 나눠** 균등 배분(초과 불가).

## 설계 (확정)

### ① 장 개시 예수금 baseline 스냅샷
- **09:00 장개시 스케줄러 잡**에서 `ord_psbl_cash`를 하루 1회 캡처.
- DB 일자별 저장 → **서버 재기동에도 유지**, **idempotent**(당일 값 있으면 재캡처 안 함).
- 비거래일/잔고 조회 실패 시 캡처 생략(가드).

### ② 레짐별 일일 투입예산률 (DB 설정 — 조정 가능)
| 레짐 SET | 예산률 | 근거 |
|---|---|---|
| Risk On 모멘텀형 | **0.90** | 상승장 과감 투입, 10% 버퍼 |
| 중립 표준형 | **0.80** | 균형 |
| 리스크오프 방어형 | **0.50** | 자본 보존 |
| 변동성 생존형 | **0.30** | 극소 노출 |

현금 버퍼는 **예산률에 내재**(1−예산률). 별도 버퍼 없음.

### ③ 종목당 매수금액 = baseline × 예산률 ÷ max_positions
- `position_size_amount = baseline_deposit × budget_rate / max_positions`
- 모든 슬롯 동일 비중, 레짐 합계가 곧 예산률(≤100%).

| 레짐 | max_pos | 종목당 | 총 투입 |
|---|---|---|---|
| Risk On | 12 | 7.5% | 90% |
| 중립 | 7 | 11.4% | 80% |
| 방어 | 4 | 12.5% | 50% |
| 변동성 | 2 | 15% | 30% |

### ④ 누적 투입 상한 가드 (order_preflight)
- 당일 누적 체결 매수금액이 `baseline × 예산률` 도달 시 **신규매수 차단**.
- baseline 기준이라 **장중 매도로 현금이 풀려도 예산 초과 안 함**.

### ⑤ 안전 clamp
- 산출 매수금액이 실제 `ord_psbl_cash`보다 크면 **작은 쪽** 사용(주문거부 방지).
- baseline 미캡처/0이면 **기존 방식(남은현금 기반)으로 폴백** — 데이터 결손이 매수를 막지 않음.

## 변경 파일

| 파일 | 변경 |
|---|---|
| `db.py` / settings | regime SET `daily_budget_rate` + baseline 테이블 스키마 |
| 신규 `daily_capital.py` | baseline 캡처/조회(일자별, 재기동 유지, idempotent) |
| `scheduler.py` | 09:00 장개시 baseline 캡처 잡 |
| `order_executor.py` | `_calc_qty`/사이징을 baseline 균등배분으로 + clamp + 폴백 |
| `order_preflight.py` | 누적 투입예산 상한 가드 |
| `tests/unit/` | 사이징·예산상한·idempotent·clamp·폴백 테스트 |

## 안전장치 (요약)
- 예산률 ≤100% + clamp + 누적가드 → **100% 초과/주문거부 원천 차단**.
- baseline DB 저장 → 재기동 견고.
- baseline 결손 시 기존 로직 폴백 → 매수 중단 없음.

3. baseline 캡처 = 09:00 장개시 잡(재기동 무관).
