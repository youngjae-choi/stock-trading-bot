# 2026-05-08 운영 콘솔 상태 정리

이 문서는 사용자가 보낸 4일 운영 후 수정/확인 요청을 기준으로, **현재 반영된 것 / 아직 안 된 것 / 확인된 이상점 / 아직 확인 안 된 것 / 답변된 질문 / 아직 답변 안 된 질문**을 다음 세션에서 바로 이어볼 수 있게 정리한 메모다.

---

## 0. 현재까지 확인한 공통 원칙

- 백엔드는 FastAPI + classic JS 콘솔 구조다.
- 화면은 React가 아니라 `backend/static/console.html` + `backend/static/js/*.js` 조합이다.
- 오늘 날짜는 **KST 기준**으로 판단한다.
- “청산”은 현재 기준 **전량 청산(일괄매도)** 의미로 정리했다.
- 일부 화면은 아직도 백엔드 truth를 그대로 보여주기보다 프론트에서 일부 해석한다.

## 1. 화면별 정리

### 1-1. Today Control

#### 반영된 것
- 화면명/네비게이션을 `Today Control`로 통일했다.
- Funnel Progress의 고정 문구 `2,500개`를 제거하고 `실시간 후보 흐름`으로 바꿨다.

#### 반영되지 않은 것
- 오늘 단계 완료/스킵/대기 로직 자체를 재설계하진 않았다.
- Today Control 전체 배치 재구성은 아직 최소 수준만 정리됐다.

#### 이상한 점 - 확인한 것
- 예전 문구처럼 화면명이 `Today's`와 `Today Control`로 섞여 보이던 문제는 정리했다.

#### 이상한 점 - 아직 확인 안 한 것
- 오후 20:50 기준으로 정말 모든 단계가 완료 상태여야 하는지.
- Today Control timeline이 실제로 왜 특정 단계 완료를 다르게 표시하는지.

#### 질문 답변됨
- “이 구조로 동작하는 게 맞냐” → 맞지만 fragile 하다고 설명함.
- “왜 화면이 별로 안 바뀐 것 같냐” → 일부는 문구/구조 정리만 했고, 큰 재설계는 아직 안 했기 때문이라고 설명함.

#### 질문 아직 답변 안 됨
- 오늘 시장톤/매매강도/Plan 상태 간의 정확한 관계.

### 1-2. Trading Monitor

#### 반영된 것
- 상단 카드 순서를 **계좌 정보 → 오늘 적용 정책**으로 유지했다.
- 청산 버튼 문구를 **전량 청산(일괄매도)** 로 바꿨다.
- 진단 화면 S9 버튼도 **즉시 전량 청산(일괄매도)** 로 맞췄다.

#### 반영되지 않은 것
- 매수 후보/보유포지션 카드의 전체 UI 재배치는 아직 최소 수정만 했다.
- 정책 문구를 더 읽기 쉬운 자연어로 다시 다듬는 작업은 남아 있다.

#### 이상한 점 - 확인한 것
- “청산”이 실제로는 `run_eod_liquidation()`으로 이어지며, 전량 매도 의미로 설계되어 있음을 확인했다.
- 다만 duplicate sell / invalid symbol / qty 0 / DB fallback 조건이 있어서 **무조건 강제 덮어쓰기**는 아니다.

#### 이상한 점 - 아직 확인 안 한 것
- 당일매수/당일매도 0원 표기가 실제 0인지, 데이터 미수집인지.
- 오늘 적용 정책 카드의 문구가 사용자가 기대한 “당일 청산 우선” 설명과 완전히 일치하는지.

#### 질문 답변됨
- “청산 = 일괄매도인데 왜 안 되냐” → 백엔드 의미는 전량 청산이 맞지만, 안전장치 때문에 일부는 스킵될 수 있다고 설명함.

#### 질문 아직 답변 안 됨
- “장이 마감되면 가지고 있는 걸 다 털어버려야 한다”를 어떤 조건까지 강제할지.

### 1-3. Trade History

#### 반영된 것
- hidden summary 카드(숫자 요약 블록)를 제거했다.
- 빈 상태/로딩 상태의 colspan을 7 → 9로 맞췄다.
- 설명을 **읽기 전용 조회 화면**으로 명확히 했다.
- `trading_orders` 기준 전체 주문 이벤트(접수/체결/실패/취소/청산 사유)를 보는 화면임을 표시했다.

#### 반영되지 않은 것
- 페이지당 20개만 보이게 하는 페이지네이션
- 시작일 ~ 종료일 기간검색 UI
- 최대 3개월 제한
- 우측 상단 조회 건수 표시

#### 이상한 점 - 확인한 것
- 과거에는 hidden summary가 남아 있었고, 테이블 구조와 colspan이 맞지 않았다.

#### 이상한 점 - 아직 확인 안 한 것
- 종목명 / 가격 / Risk Profile / 청산사유가 왜 충분히 남지 않는지.
- 주문/매도 로그가 backend에 제대로 저장되는지.

#### 질문 답변됨
- “지금 보면 되는가?” → 보긴 되지만, 눈에 확 띄는 큰 재배치는 아니라는 점을 설명함.

#### 질문 답변됨
- “제대로 매수 매도 로그를 남기고 이를 표시하고 있는가?” → `trading_orders` 기준으로 주문 접수/실패/취소/청산 사유를 표시하며, `/api/v1/orders/range`는 `history_scope=all_order_events`를 반환한다.

## 2. 공통 이슈

### 반영된 것
- 종료/청산 관련 문구는 전량 청산 의미로 통일했다.
- Today Control / Trade History / Diagnostics / Funnel의 일부 trust-breaker는 정리했다.

### 반영되지 않은 것
- 장 종료 후 과거 데이터를 지우지 않고 유지하는 정책의 완전한 구현
- 주식투자전문 AI 추천값 자동 반영

### 이상한 점 - 확인한 것
- 프론트가 backend truth를 직접 보여주기보다 일부 해석한다.
- 그래서 새로고침/캐시/스크립트 상태에 따라 “변한 게 없어 보임”이 생길 수 있다.

### 이상한 점 - 아직 확인 안 한 것
- 5월 4일 5%, 6일 7% 급등장인데 시스템 수익이 1%대인 이유.
- 매매 로직 자체가 왜 약한지.

### 질문 답변됨
- “구조가 이대로 동작하냐” → 네, classic script shell로 동작한다.
- “왜 UI가 별로 안 바뀐 것 같냐” → 큰 재설계가 아니라 trust-first 정리만 했기 때문이다.

### 질문 아직 답변 안 됨
- 정상 시스템이 어떻게 동작해야 하는지에 대한 최종 정의.
- 더 좋은 프로세스 / 불필요한 프로세스 판정.

## 5. 미답변 질문 분류표

아래는 아직 결론이 안 난 항목을 **코드 수정으로 해결할 것인지**, **PM 질문이 필요한지**로 나눈 목록이다.

### 5-1. 코드 수정/확인 필요

- **Today Control / Daily Plan / Funnel / Diagnostics 상태 불일치**
  - 화면마다 해석 레이어가 달라 보이는 문제를 정리해야 한다.
- **Trade History 저장/표시 흐름**
  - 주문/매도 로그가 어디까지 저장되고, 화면에 무엇을 보여줄지 확인이 필요하다.
- **Knowledge 재로딩 / 정책 반영 흐름**
  - 과거 전략을 다시 불러오고 적용하는 실제 경로를 연결해야 한다.
- **Missed Entries / False Positive 데이터 축적**
  - 어떤 source of truth를 표시할지 정해야 한다.
- **Trade Review 저장 위치**
  - MD 파일인지 DB인지, 저장/불러오기 규칙을 확인해야 한다.
- **Settings 추천값 반영 방식**
  - UI 통일과 함께 추천값을 자동 반영할지, 승인 후 반영할지 구현이 필요하다.
- **빈 카드/빈 화면 표현**
  - 실제 데이터 없음인지, 미수집인지, 실패인지 구분해서 표시해야 한다.

### 5-2. PM 질문이 필요한 항목

- **장 종료 후 보관 정책**
  - 과거 데이터를 지우지 않고 유지할지, 어떤 범위까지 보관할지.
- **AI 추천값 반영 정책**
  - 추천값을 자동 적용할지, 항상 승인 후 적용할지.
- **오늘 상태 판정 기준**
  - 20:50 이후 Today Control / Plan / Market Tone / trade strength를 어떤 규칙으로 보여줄지.
- **Missed Entries의 포함 범위**
  - S3 탈락 전부를 보여줄지, 일부 조건만 보여줄지.
- **Settings 추천 기준**
  - 어떤 데이터가 모이면 추천값을 갱신할지.

## 6. 화면별 수정 사항 목록

### 완료/반영됨

- **Today Control**: 화면명 통일, Funnel Progress 고정 숫자 제거.
- **Trading Monitor**: 청산 문구를 전량 청산(일괄매도)로 통일.
- **Trade History**: hidden summary 제거, 빈 상태/로딩 colspan 정리, 읽기 전용 문구 정리.
- **Funnel Monitor**: `2,500개` 문구 제거, total universe trust-breaker 완화.
- **System Diagnostics**: S1~S11 실제 실행결과 중심으로 정리, last_run 계약 추가.

### 아직 남은 것

- **Today Control**: timeline/단계 상태의 실제 의미 정리.
- **Trading Monitor**: 카드 배치와 정책 문구의 자연어화.
- **Daily Plan**: LLM 분석요약 / Profile Pack / 제외종목 카드 위치 조정.
- **Funnel Monitor**: 카드 그룹 재배치, Profile 배정현황 정리.
- **Knowledge**: 과거 이력 재로딩, 정책 반영 기능.
- **Missed Entries**: S3 탈락 종목 표시 구조.
- **False Positive**: 손실 원인 축적 및 검색 기능.
- **Trade Review**: 저장/불러오기, MD 파일 유지 여부.
- **System Diagnostics**: 빈 카드 제거 및 상태 가시성 개선.
- **Settings**: 임계치 UI 통일, 추천값 반영 방식 정리.

## 7. 사용자에게 보낼 질문안

아래 질문만 답이 정해지면 남은 구현 방향이 바로 정리된다.

1. **장 마감 후 과거 데이터는 어디까지 남겨둘까요?**
   - 화면에서 숨길지, 그대로 보관할지, 별도 아카이브로 둘지 정해야 한다.
2. **AI 추천값은 자동 반영할까요, 승인 후 반영할까요?**
   - Settings / Knowledge / Daily Plan 전반의 동작이 이 기준을 따른다.
3. **빈 화면 표시는 어떤 문구로 구분할까요?**
   - `데이터 없음`, `미수집`, `실패`를 구분해서 보여줄지 결정이 필요하다.
4. **Today Control / Daily Plan / Funnel / Diagnostics의 최종 기준 화면은 어디로 둘까요?**
   - 같은 상태를 여러 화면이 다르게 해석하는 문제를 정리해야 한다.
