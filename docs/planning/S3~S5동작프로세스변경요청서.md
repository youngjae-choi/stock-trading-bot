S3~S5 동작프로세스 변경요청서
1. 문서 목적
본 문서는 Dantabot 자동매매 시스템의 핵심 의사결정 구간인 S3 유니버스 필터, S4 하이브리드 스크리닝, S5 Daily Trading Plan 생성 단계의 동작 방식을 변경하기 위한 요청서이다.

기존 시스템은 당일 시장 분석과 후보 종목 데이터를 중심으로 S3~S5를 실행하고, S5에서 AI가 매일 RulePack을 생성하는 구조였다.

변경 후 시스템은 다음 원칙을 따른다.

1. S5에서 매일 RulePack 전체를 새로 생성하지 않는다.
2. 고정 Base RulePack과 Risk Profile 4종은 유지한다.
3. 매일 생성하는 것은 Daily Trading Plan이다.
4. S3~S5는 당일 시장 데이터뿐 아니라 Review & Audit 학습 결과와 외부 전문가 지식베이스를 참고한다.
5. 기억은 LLM이 아니라 DB와 Backend가 담당한다.
6. AI는 매일 필요한 기억과 지식을 프롬프트로 전달받고 판단한다.
7. 모든 결과는 Global Risk Guard 검증을 통과해야 한다.
2. 기존 S3~S5 구조
기존 구조는 다음과 같다.

S1 KIS 토큰 갱신
→ S2 시장 톤 분석
→ S3 유니버스 필터
→ S4 하이브리드 스크리닝
→ S5 RulePack 자동 생성
→ S6 Decision Engine 활성화
기존 문제점:

- S5에서 매일 RulePack 전체를 새로 생성하여 전략 일관성이 떨어짐
- Review & Audit 결과가 다음날 S3~S5에 구조적으로 반영되지 않음
- 외부 전문가 자료나 학습자료가 종목 선정/전략 생성에 반영되지 않음
- LLM의 판단이 당일 데이터에 과도하게 의존함
- AI가 리스크 파라미터까지 임의 조정할 위험이 있음
- 종목별 Risk Profile 배정 근거가 약함
- 복기 결과와 실제 다음날 매매 의사결정이 연결되지 않음
3. 변경 후 전체 구조
변경 후 구조는 다음과 같다.

S1 KIS 토큰 갱신
→ S2 시장 톤 분석
→ S3 유니버스 필터
→ S4 하이브리드 스크리닝

→ S10 Review & Audit 생성

→ 다음 거래일 S3~S5에 반영
핵심 변경:

기존: S5 RulePack 자동 생성
변경: S5 Daily Trading Plan 생성
4. 핵심 설계 원칙
4.1 AI가 기억하지 않는다
LLM은 세션이 바뀌거나 Provider가 바뀔 수 있으므로 기억 주체가 되어서는 안 된다.

기억 주체는 다음과 같다.

- DB
- Backend
- Learning Memory Module
- Expert Knowledge Base
- Market Context Memory
AI는 S3~S5 실행 시 Backend가 제공하는 구조화된 컨텍스트를 참고하여 판단한다.

4.2 S3~S5는 4종 컨텍스트를 함께 참고한다
S3~S5 실행 시 아래 정보를 함께 사용한다.

1. 당일 시장 데이터
2. S2 시장 톤 분석 결과
3. Review & Audit에서 생성된 Learning Memory
4. 외부 전문가 자료에서 추출된 Expert Knowledge

5. 신규 모듈 추가 요청
5.1 Learning Memory Module
Review & Audit에서 생성된 복기 결과를 다음날 S3~S5에서 사용할 수 있도록 구조화하여 저장하는 모듈이다.

역할:

- 일일 복기 결과 저장
- Risk Profile별 성과 저장
- 청산 사유별 성과 저장
- 트레일링 품질 분석 저장
- S4 후보 선정 품질 저장
- S5 Daily Plan 품질 저장
- 다음날 S3/S4/S5용 요약 컨텍스트 생성
처리 흐름:

장 종료
→ S10 Review & Audit 생성
→ 외부 LLM 또는 내부 LLM으로 복기
→ S11 Learning Memory Builder 실행
→ 구조화된 학습 메모리 DB 저장
→ 다음날 S3/S4/S5 프롬프트에 필요한 항목만 주입
5.2 Expert Knowledge Base
주식 전문가 강의, 학습자료, 사용자 전략 메모, 시장 해설 자료 등을 AI가 분석하여 S3~S5에서 참고할 수 있도록 구조화하는 모듈이다.

처리 흐름:

외부자료 업로드
→ 텍스트 추출
→ AI 요약
→ 핵심 전략 원칙 추출
→ S3/S4/S5 적용 위치 태깅
→ 위험도 및 신뢰도 평가
→ 사용자 승인
→ Knowledge Base 저장
→ S3/S4/S5 프롬프트에 주입
→ Review & Audit에서 성과 추적
중요 원칙:

외부자료는 직접 매매 룰로 적용하지 않는다.
외부자료는 구조화된 전략 지식으로 변환한 뒤 참고자료로만 사용한다.
승인된 지식만 S3~S5에 주입한다.
Global Risk Guard를 완화하는 지식은 무시한다.
Risk Profile 원본 값 변경은 자동으로 하지 않는다.
6. 기억의 종류
시스템은 최소 3종류의 기억을 관리해야 한다.

6.1 Trade Learning Memory
실제 매매 결과와 복기에서 나온 학습이다.

예시:

- 최근 THEME_SPIKE 손실 변동성이 높음
- HIGH_VOL 트레일링 수익 반납률이 높음
- MID_VOL이 최근 가장 안정적인 성과를 보임
- 후보 21개 중 실제 진입 0건으로 진입 전환율 낮음
출처:

- Review & Audit
- 주문/체결 데이터
- 포지션 상태 데이터
- Daily Plan 결과
6.2 Expert Knowledge Memory
외부자료에서 추출한 전문가 지식이다.

예시:

- 거래량 급증 직후 첫 장대양봉 추격은 위험하다.
- VWAP 위 눌림 후 거래량이 재증가하는 종목이 안정적이다.
- 거래대금 없는 급등주는 체결 리스크가 크다.
- 뉴스 근거가 약한 우선주성 급등은 보수적으로 처리한다.
출처:

- 전문가 강의
- 학습자료
- 리포트
- 사용자 전략 메모
- 외부 시장 해설 자료
6.3 Market Context Memory
최근 시장 상황과 테마 흐름에서 나온 단기 기억이다.

예시:

- 최근 반도체 강세
- 전력기기 테마 강세
- 2차전지 변동성 확대
- 코스닥 변동성 증가
- 미국 기술주 강세
출처:

- S2 시장 톤 분석
- 뉴스/테마 분석
- 지수 흐름
- 거래대금 상위 종목
7. S3 유니버스 필터 변경 요청
7.1 S3의 역할
S3는 전체 종목에서 1차 후보군을 추리는 단계이다.

기존:

전체 종목
→ 기본 거래대금/시총/거래정지/관리종목 필터
→ Layer 1 Universe 생성
변경 후:

전체 종목
→ 기본 필터
→ Review Memory 기반 제외/선호 힌트 반영
→ Expert Knowledge 기반 제외/선호 힌트 반영
→ Market Context 기반 섹터/테마 힌트 반영
→ Layer 1 Universe 생성
7.2 S3 입력 데이터
S3는 아래 데이터를 입력으로 받아야 한다.

{
  "raw_universe": "전체 종목 데이터",
  "market_context": "S2 시장 톤 및 테마 요약",
  "s3_learning_memory": "최근 유니버스 필터 관련 복기 요약",
  "s3_expert_knowledge": "외부자료에서 추출된 유니버스 필터링 원칙",
  "global_exclusion_rules": "관리종목, 거래정지, 투자경고 등 시스템 필터"
}
7.3 S3에 반영할 Review Memory
S3는 Review & Audit에서 추출된 아래 유형의 학습을 참고한다.

- 최근 제외했으나 급등한 종목군
- 후보에 넣었으나 반복적으로 손실이 발생한 종목군
- 유동성 부족으로 체결 품질이 나빴던 종목군
- 특정 테마에서 손실이 반복된 경우
- 특정 시장 톤에서 너무 많은 후보 또는 너무 적은 후보가 나온 경우
예시:

{
  "for_step": "S3_UNIVERSE_FILTER",
  "recent_filter_lessons": [
    {
      "lesson": "거래대금이 낮은 급등주는 체결 리스크가 높았다.",
      "action_hint": "거래대금 부족 종목은 제외 우선순위를 높인다.",
      "strength": "high"
    },
    {
      "lesson": "positive 시장에서 후보 수가 부족하면 실제 진입 기회가 낮았다.",
      "action_hint": "시장 톤이 positive 이상이면 유동성 좋은 중형주 후보를 충분히 확보한다.",
      "strength": "medium"
    }
  ]
}
7.4 S3에 반영할 Expert Knowledge
S3는 외부자료에서 추출된 아래 지식을 참고한다.

- 유동성 낮은 종목 제외
- 우선주성 급등주 보수 처리
- 거래대금 증가 종목 우선
- 시장 주도 섹터 우선
- 관리/투자경고/거래정지 종목 제외
- 호가 공백이 큰 종목 제외
S3 프롬프트 삽입 예시:

[Expert Knowledge for S3 Universe Filtering]

다음은 승인된 전문가 지식베이스에서 추출된 유니버스 필터링 참고 원칙입니다.

1. 유동성이 낮은 급등주는 체결 리스크가 크므로 제외 우선순위가 높습니다.
2. 거래대금이 충분하고, 시장 주도 섹터에 속한 종목을 우선 고려하십시오.
3. 투자경고, 관리종목, 뉴스 근거가 약한 우선주성 급등주는 보수적으로 처리하십시오.

위 내용은 참고 원칙이며, 시스템의 Global Risk Guard와 기존 필터 조건을 위반할 수 없습니다.

8. S4 하이브리드 스크리닝 변경 요청
8.1 S4의 역할
S4는 S3에서 생성된 유니버스 중에서 실제 감시 후보를 선별하는 단계이다.

기존:

Layer 1 Universe
→ 기술적 지표, 거래대금, AI 정성 평가
→ Layer 2 후보 생성
변경 후:

Layer 1 Universe
→ 기술적 지표
→ 거래대금/거래량
→ 테마/뉴스
→ Review Memory 기반 최근 성공/실패 패턴 반영
→ Expert Knowledge 기반 패턴 선호/회피 반영
→ Market Context 기반 당일 테마 가중
→ Layer 2 후보 생성
8.2 S4 입력 데이터
{
  "layer1_universe": "S3 결과",
  "market_tone": "positive",
  "theme_context": ["반도체", "전력기기"],
  "technical_features": "가격, 거래량, VWAP, 이동평균, 변동성 등",
  "news_theme_features": "뉴스, 테마, 재료",
  "s4_learning_memory": "최근 스크리닝 성공/실패 패턴",
  "s4_expert_knowledge": "전문가 지식 기반 패턴 원칙"
}
8.3 S4에 반영할 Review Memory
S4는 아래 복기 내용을 참고한다.

- 후보 수는 충분했으나 실제 진입 전환율이 낮았던 경우
- 특정 패턴의 후보가 반복 손실을 만든 경우
- 제외 종목 중 급등한 종목이 있었던 경우
- THEME_SPIKE 후보가 과다 선별되어 손실을 만든 경우
- VWAP, 거래량, 눌림 패턴 조건이 너무 빡세거나 느슨했던 경우

      "strength": "medium"
    },
    {

    "시장 주도 섹터 내 대장주 또는 2등주"
  ]

8.4 S4에 반영할 Expert Knowledge
S4에서 외부 지식은 가장 크게 활용된다.

선호 패턴:

- VWAP 위 눌림 후 재상승
- 거래량 급증 후 거래량 재증가
- 시장 주도 섹터 내 대장주 또는 2등주
- 과열 추격이 아닌 눌림 재상승
- 거래대금 증가와 가격 안정이 동시에 나타난 종목
회피 패턴:

- 첫 급등 장대양봉 추격
- 거래대금 부족 급등주
- 뉴스 근거 약한 우선주성 급등
- 고점 부근 거래량만 급증한 종목
- 이미 과열된 테마 후발주
S4 프롬프트 삽입 예시:

[Expert Knowledge for S4 Hybrid Screening]

최근 승인된 전문가 지식에 따르면, 다음 패턴을 우선 고려하십시오.

선호 패턴:
- VWAP 위 눌림 후 재상승
- 거래량 급증 후 거래량 재증가
- 시장 주도 섹터 내 대장주 또는 2등주
- 장대양봉 이후 과열이 아니라 눌림을 만든 종목

회피 패턴:
- 첫 급등 장대양봉 추격
- 거래대금이 부족한 급등주
- 뉴스 근거가 약한 우선주성 급등
- 고점 부근에서 거래량만 급증한 종목

후보 선정 사유에는 어떤 전문가 지식이 적용되었는지 간단히 명시하십시오.

9. S5 Daily Trading Plan 생성 변경 요청
9.1 S5의 역할 변경
기존 S5:

S5 - RulePack 자동 생성
08:45 KST - LLM → rulepacks
변경 후 S5:

S5 - Daily Trading Plan 생성
08:45 KST - LLM → daily_trading_plan
S5는 더 이상 매일 RulePack 전체를 생성하지 않는다.

S5의 역할은 다음과 같다.

- 오늘의 시장 톤과 매매 강도 결정
- S4 후보 중 최종 감시 종목 결정
- 종목별 Risk Profile 배정
- 종목별 진입 정책 설정
- 종목별 제외/주의 사유 작성
- Daily Override 생성
- S6 Decision Engine이 사용할 Daily Trading Plan 작성
9.2 고정 RulePack과 Risk Profile 유지
S5가 변경할 수 없는 것:

- Base RulePack
- Risk Profile 원본 값
- Global Risk Guard
- 일일 손실 한도
- 주간 손실 한도
- 월간 손실 한도
- 최대 보유 종목 수
- 종목당 최대 비중 상한
- 장마감 강제청산 정책
- 손절선 하향 금지 정책
- 고정 익절 OFF 원칙
S5가 생성할 수 있는 것:

- Daily Trading Plan
- 오늘 매매 강도
- 오늘 후보 종목
- 종목별 Risk Profile 배정
- entry_policy
- max_theme_spike_positions
- min_ai_confidence
- volume_filter_multiplier
- 신규매수 허용 여부
- 제외 종목 목록
- 주의 테마/종목
9.3 Risk Profile 4종
S5는 각 종목에 아래 4개 중 하나의 Risk Profile을 배정한다.

LOW_VOL
MID_VOL
HIGH_VOL
THEME_SPIKE
배정 기준:

LOW_VOL:
- 대형주
- 저변동성
- 유동성 충분
- 추세 안정

MID_VOL:
- 일반적인 단타 후보
- 중간 변동성
- 거래대금 양호
- 시장 주도 섹터 내 안정 후보

HIGH_VOL:
- 최근 변동성 확대
- 상승/하락폭 큼
- 테마 영향 있음
- 진입 신뢰도 높은 경우만 허용

THEME_SPIKE:
- 당일 급등 테마주
- 뉴스/재료 기반 급등주
- 과열 가능성 높음
- 비중과 개수 제한 필요

9.5 S5에 반영할 Expert Knowledge
S5는 외부 지식을 종목별 전략과 Profile 배정에 활용한다.

활용 예시:

- 첫 급등 추격형 종목은 THEME_SPIKE 또는 제외
- 거래량 급증 후 과열된 종목은 wait_for_pullback 정책 적용
- 뉴스 근거 약한 급등주는 allowed=false 또는 낮은 confidence
- 시장 주도 섹터 내 대장주는 우선 감시 후보

3. 거래량 급증 후 이미 과열된 종목은 entry_policy를 wait_for_pullback으로 설정하십시오.
4. 뉴스 근거가 약한 급등주는 allowed=false 또는 confidence를 낮게 설정하십시오.
5. 전문가 지식은 참고자료이며, Risk Profile 원본 값은 변경하지 마십시오.

10. 반영 수준 정책
Review Memory와 Expert Knowledge의 반영 수준은 3단계로 나눈다.

10.1 Level 1: Context Only
참고만 하는 수준이다.

- 프롬프트에 참고자료로 제공
- 점수 또는 룰 값을 직접 변경하지 않음
- 기본값
예시:

최근 THEME_SPIKE 손실 변동성이 높았다.
전문가 자료에서는 첫 급등 추격을 경계하라고 한다.
10.2 Level 2: Daily Plan Override
하루짜리 운용 계획에만 반영하는 수준이다.

허용 예시:

- THEME_SPIKE 허용 개수 축소
- 신규매수 강도 낮추기
- min_ai_confidence 상향
- volume_filter_multiplier 상향
- 특정 종목 제외
- 특정 종목을 더 보수적인 Profile로 배정
예시:

{
  "daily_overrides": {
    "max_theme_spike_positions": 1,
    "min_ai_confidence": 0.70,
    "volume_filter_multiplier": 2.2
  }
}
10.3 Level 3: Profile Version Change
Risk Profile 원본 값을 변경하는 수준이다.

예시:

HIGH_VOL trailing_stop_rate 0.05 → 0.045
THEME_SPIKE max_position_rate 0.05 → 0.03
MID_VOL trailing_activate_profit 0.025 → 0.02
이 단계는 자동 반영 금지이다.

반드시 아래 흐름을 거친다.

AI 복기 제안
→ Strategy Adjustment Candidate 생성
→ 사용자 승인
→ profile-v1.1 생성
→ 이후 적용

11.1 자동 반영 가능

12. 검증 정책
S3~S5 결과는 반드시 검증을 통과해야 한다.

12.1 S3 검증
- 관리종목/거래정지 종목 제외 여부
- 거래대금 기준 충족 여부
- 유동성 부족 종목 과다 포함 여부
- Expert Knowledge 적용 근거 존재 여부
- S3 출력 Schema 검증
12.2 S4 검증
- 후보 점수 범위 검증
- 후보 선정 사유 존재 여부
- 위험 패턴 누락 여부
- knowledge_refs 유효성 검증
- memory_refs 유효성 검증
- THEME_SPIKE 후보 과다 여부
- S4 출력 Schema 검증
12.3 S5 검증
- 모든 symbol_assignments에 profile 존재 여부
- profile 값이 LOW_VOL/MID_VOL/HIGH_VOL/THEME_SPIKE 중 하나인지 검증
- allowed=false 종목은 매수 대상에서 제외되는지 검증
- Daily Override가 Global Risk Guard를 위반하지 않는지 검증
- Risk Profile 원본 값 변경 시도 여부 검증
- 고정 익절 OFF 유지 여부 검증
- 장마감 강제청산 ON 유지 여부 검증
- Daily Plan Schema 검증

23. 최종 요약
이번 변경의 핵심은 다음과 같다.

S3~S5는 더 이상 당일 데이터만 보고 판단하지 않는다.
Review & Audit에서 나온 실제 매매 학습 결과와 외부 전문가 지식을 함께 참고한다.
단, 기억은 LLM이 아니라 DB와 Backend가 담당한다.
AI는 S3~S5 실행 시 필요한 구조화 컨텍스트를 전달받아 판단한다.
S5는 RulePack을 새로 만드는 것이 아니라 Daily Trading Plan을 생성한다.
Daily Trading Plan은 오늘 어떤 종목에 어떤 Risk Profile을 적용할지 결정한다.
외부자료와 복기 결과는 자동으로 리스크 한도를 완화할 수 없다.
모든 결과는 Global Risk Guard, Schema Validation, Audit Log를 통과해야 한다.
한 줄 결론:

Dantabot은 매일의 복기에서 배우고, 외부 전문가 지식에서 원칙을 배우되, 실제 매매 반영은 항상 구조화·승인·검증을 거쳐 S3~S5에 반영해야 한다.
