# Python 백엔드 시스템 개발기획 v2

## 1. 문서 목적과 현재 단계

이 문서는 KIS Open API 기반 단타 자동매매 시스템의 **Python 백엔드 개발 방향**을 PM 관점에서 이해할 수 있도록 정리한 기획 문서다.  
기존 `시스템기획서.md`가 운영 콘솔과 화면 관점에 초점을 두었다면, 본 문서는 그 화면 뒤에서 실제로 돌아갈 **백엔드 엔진, API, 저장 구조, 운영 사이클, 개발 순서**를 정의하는 것이 목적이다.

현재 단계는 **구현 전 상세 설계 단계**다.

- 아직 목표 구조를 확정하는 문서 단계이며, 구현 완료 문서가 아니다.
- 기존 문서의 핵심 원칙을 유지한다.
- 특히 아래 원칙을 백엔드 설계의 기본 전제로 삼는다.
  - AI는 해석과 RulePack 생성 담당
  - 시스템은 정량 계산, 조건 평가, 주문 실행 담당
  - 학습은 장중 실시간 자동 변경이 아니라 **장마감 후 분석 후 다음날 반영**
  - RulePack은 **기계용 구조화 데이터**와 **사용자용 자연어 요약**을 함께 저장
  - 유니버스 생성은 종목 단건 순회가 아니라 **KIS 유니버스/랭킹 API 우선**
  - KIS 신규 고객 호출 제한(실전 3일간 3 TPS)을 고려한 보수적 호출 제어가 필요

이 문서는 `ONBOARDING.md`, `AGENTS.md`, `DOC_HIERARCHY.md`, `IMPLEMENTATION_RULES.md`, `TEST_RULES.md` 및 `docs/reference`, `docs/planning` 기존 문서를 종합해 작성한다.

## 2. 핵심 결론

### 2.1 한 줄 결론

이 시스템의 백엔드는 **Python 기반 이벤트형 자동매매 엔진 + `/api/v1/bot` 운영 API + 장마감 학습 저장소** 구조로 가야 한다.

### 2.2 PM이 먼저 이해해야 할 핵심 포인트

1. 이 시스템은 "주문 버튼 앱"이 아니라 **자동으로 판단하고, 사람은 감시하는 관제형 시스템**이다.
2. 따라서 백엔드의 1순위는 화려한 기능이 아니라 **안전한 실행, 차단, 기록, 복기**다.
3. 유니버스는 전체 종목을 무식하게 훑는 방식이 아니라, KIS의 **랭킹/조건검색 API로 먼저 좁히고**, 필요한 후보만 단건 조회해야 한다.
4. 주문 판단은 하나의 모델이 아니라 여러 모듈이 나눠 맡아야 한다.
   - 데이터 수집
   - 룰 해석
   - 진입 판단
   - 리스크 차단
   - 주문 실행
   - 기록/학습
5. 학습 기능은 중요하지만, 장중에 스스로 전략을 바꾸게 하면 위험하다.
   - 장중: 이미 승인된 RulePack만 실행
   - 장마감 후: 결과 분석
   - 다음 거래일: 검증된 개선안만 반영
6. 운영 API는 단순 데이터 조회용이 아니라, "왜 주문했는가", "왜 차단했는가", "오늘 룰이 무엇인가"를 설명할 수 있어야 한다.

## 3. Python 백엔드 목표 아키텍처

### 3.1 목표 구조

백엔드는 아래 4계층으로 설계한다.

1. **외부 연동 계층**
   - KIS REST API
   - KIS WebSocket
   - 뉴스/테마/보조 데이터 소스
   - 단타봇 API 허브와의 연동 채널
2. **실행 엔진 계층**
   - Universe Engine
   - RulePack Engine
   - Decision Engine
   - Risk & Order Guard
   - Scheduler/Realtime
3. **운영 API 계층**
   - `/api/v1/bot/*`
   - 운영 콘솔, 관리자 화면, 외부 허브가 읽는 표준 API
4. **저장/감사 계층**
   - 실시간 상태 저장
   - 주문/판단/Audit 로그 저장
   - 학습용 스냅샷 저장
   - 장마감 리포트 저장

### 3.2 권장 데이터 흐름

```text
외부 데이터/KIS API
    ↓
KIS Gateway + RateLimiter
    ↓
Universe Engine
    ↓
RulePack Engine
    ↓
Decision Engine
    ↓
Risk & Order Guard
    ↓
KIS 주문 실행
    ↓
Audit/Learning Store 저장
    ↓
/api/v1/bot 운영 API 제공
```

### 3.3 설계 원칙

- 엔진과 UI를 분리한다. 화면이 꺼져도 자동매매 엔진은 계속 돌아야 한다.
- 실전 주문 전에는 반드시 Risk & Order Guard를 통과해야 한다.
- 모든 중요한 판단은 "결과"뿐 아니라 "근거"까지 저장한다.
- RulePack은 자연어 설명과 machine rules를 동시에 보관한다.
- REST와 실시간(WebSocket)을 혼합하되, 실시간 데이터 이상 시 안전하게 주문을 막을 수 있어야 한다.

### 3.4 권장 런타임 구분

- **Batch 영역**: 장 시작 전 유니버스 생성, RulePack 검증, 장마감 복기
- **Realtime 영역**: 체결/호가/진입 조건 감시, 주문/청산
- **API 영역**: 운영 콘솔 조회, 상태 확인, 수동 긴급 제어

## 4. 모듈 설계

### 4.1 KIS Gateway

역할:

- KIS REST/WebSocket와 통신하는 단일 관문
- 토큰 발급/갱신
- 요청 헤더, 트랜잭션 ID, 응답 표준화
- KIS 오류 코드 표준 해석

필수 책임:

- 실전/모의 환경 구분
- 주문/조회 API 경로 분리
- `msg_cd`, `rt_cd`, HTTP 상태를 공통 포맷으로 변환
- 장애 시 상위 모듈에 "재시도 가능/불가" 신호 전달

PM 관점 의미:

- KIS 연동이 여기로 모이면, 나중에 API 변경이나 장애 대응을 한 곳에서 관리할 수 있다.

### 4.2 RateLimiter

역할:

- KIS 호출 제한을 넘지 않도록 요청 속도를 제어

설계 원칙:

- 신규 실전 계정은 3일간 3 TPS 제한을 기준으로 더 보수적으로 시작
- 모의/기존 계정/실전 일반 구간은 정책별 프로필 분리
- 단일 전역 제한이 아니라 **API 그룹별 제한 + 계정별 제한 + 환경별 제한**을 고려

필수 기능:

- 초당 호출량 제어
- 백오프 및 재시도
- `EGW00201` 감지 시 즉시 감속
- 현재 제한 상태를 운영 API로 노출

권장 운영 프로필:

- 실전 신규 3일 이내: 목표 2 TPS
- 실전 일반: 목표 10~15 TPS 범위에서 보수 운영
- 모의투자: 별도 프로필 운영

### 4.3 Universe Engine

역할:

- 전체 시장에서 "오늘 감시할 종목 후보군"을 만든다.

핵심 방향:

- 종목 마스터 전체를 단건 순회하지 않는다.
- KIS 랭킹/조건검색 API를 먼저 사용해 **Top N 후보군**을 만든다.

우선 적용 체인:

1. 랭킹 기반 유니버스
   - 거래량 순위
   - 시가총액 순위
   - 체결강도/거래대금 관련 순위
2. 조건검색 기반 유니버스
   - 사용자 전략에 맞는 조건검색식 결과 사용
3. 보강 필터
   - 투자자 수급
   - 외국인/기관 동향
   - 차트 기반 계산 지표
4. 최종 단건 보강
   - 현재가
   - 호가
   - 체결 정보

출력:

- `universe_candidates`
- `universe_source_api`
- 각 후보의 생성 근거
- API 호출 수/실패 수

### 4.4 RulePack Engine

역할:

- AI가 만든 전략 초안을 백엔드가 실행 가능한 형태로 정리하고 검증한다.

핵심 원칙:

- RulePack은 두 개를 함께 저장한다.
  - machine_rules: 시스템 실행용
  - human_summary: PM/운영자 표시용

필수 검증:

- JSON Schema 검증
- 값 범위 검증
- 리스크 한도 검증
- 금지 조건 검증
- 실행 가능성 검증

예시 검증 항목:

- `rsi_min <= rsi_max`
- 손절폭이 최대 허용 범위를 넘지 않는지
- 일일 손실한도 완화 시도 여부
- 최대 포지션 수 초과 여부

출력:

- `draft`
- `validated`
- `approved`
- `rejected`
- `applied`

### 4.5 Decision Engine

역할:

- 유니버스 후보와 RulePack을 바탕으로 실시간 매수/매도 판단을 수행한다.

판단 범위:

- 진입 가능 여부
- 진입 대기 이유
- 차단 이유
- 청산 조건 충족 여부

입력:

- 실시간 시세/호가/체결
- RulePack machine rules
- 종목별 상태
- 계좌/포지션 상태

출력:

- `WATCHING`
- `ENTRY_READY`
- `ENTRY_BLOCKED`
- `ORDER_REQUESTED`
- `POSITION_OPEN`
- `EXIT_READY`
- `EXIT_REQUESTED`

중요 원칙:

- Decision Engine은 "주문해도 된다"까지만 판단한다.
- 실제 주문 허용 여부는 항상 **Risk & Order Guard가 최종 결정**한다.

### 4.6 Risk & Order Guard

역할:

- 실전 주문 전 마지막 안전장치

반드시 막아야 하는 항목:

- 일일 손실한도 초과
- 최대 포지션 수 초과
- 종목당 비중 초과
- 주문 가능 현금 부족
- 스프레드 과다
- 실시간 데이터 지연
- KIS 장애 상태
- 중복 주문

세부 기능:

- 신규 진입 차단
- 기존 포지션 청산 허용 여부 판단
- 긴급정지(Halt) 적용
- 주문 전 최종 수량 산출

PM 관점 의미:

- 이 모듈이 강해야 "AI가 판단했다"는 이유만으로 위험한 주문이 나가지 않는다.

### 4.7 Audit/Learning Store

역할:

- 장중 판단과 장마감 학습에 필요한 모든 기록을 저장한다.

반드시 저장할 것:

- RulePack 원본과 자연어 요약
- 유니버스 생성 결과
- 종목별 판단 스냅샷
- 주문 요청/접수/체결/실패
- 차단 사유
- 미진입 후보의 이후 성과
- 청산 사유
- 하루 복기 결과

중요 원칙:

- 거래 결과만 저장하면 부족하다.
- **왜 샀는지 / 왜 안 샀는지 / 왜 막았는지**가 같이 남아야 학습이 가능하다.

### 4.8 Scheduler/Realtime

역할:

- 하루 운영 일정을 실행하고, 장중에는 실시간 감시를 지속한다.

담당 작업:

- 장 시작 전 토큰 갱신
- RulePack 적재
- 유니버스 생성
- 장중 WebSocket 연결/복구
- 시간대별 스캔
- 장마감 청산
- 장마감 복기 배치

필수 기능:

- 작업 스케줄 상태 추적
- 실패 작업 재시도
- 중복 실행 방지
- 장애 복구 후 안전 재개

## 5. API 설계 (`/api/v1/bot` 기준)

운영 API는 "데이터 조회"와 "운영 통제" 두 가지 목적을 가진다.  
기본 원칙은 **운영 콘솔과 단타봇 API 허브가 같은 표준 API를 읽게 하는 것**이다.

### 5.1 API 설계 원칙

- 읽기 API와 제어 API를 분리한다.
- 실시간 상태와 장마감 리포트를 구분한다.
- 응답에는 숫자 데이터와 자연어 설명을 함께 담는다.
- "정상/실패"뿐 아니라 "차단 이유"를 표준 필드로 제공한다.

### 5.2 핵심 조회 API

| Endpoint | 목적 |
|---|---|
| `GET /api/v1/bot/overview` | 엔진 상태, RulePack 상태, 리스크 상태, 오늘 요약 |
| `GET /api/v1/bot/rulepack/today` | 오늘 RulePack의 machine/human 요약 조회 |
| `GET /api/v1/bot/universe` | 오늘 유니버스 결과와 후보군 출처 조회 |
| `GET /api/v1/bot/candidates` | 현재 감시 후보 종목과 상태 조회 |
| `GET /api/v1/bot/decisions/live` | 실시간 진입/청산 판단 현황 조회 |
| `GET /api/v1/bot/orders` | 주문 요청/접수/체결/실패 로그 조회 |
| `GET /api/v1/bot/blocks` | 주문 차단 사유 조회 |
| `GET /api/v1/bot/risk` | 손실한도, 포지션 수, 신규 주문 가능 여부 조회 |
| `GET /api/v1/bot/data-health` | KIS REST, KIS WebSocket, DB, 스케줄러 상태 조회 |
| `GET /api/v1/bot/review/daily` | 일일 복기 결과 조회 |
| `GET /api/v1/bot/learning/summary` | 학습 메모리 요약과 개선 제안 조회 |

### 5.3 제어 API

| Endpoint | 목적 |
|---|---|
| `POST /api/v1/bot/control/halt` | 긴급정지 |
| `POST /api/v1/bot/control/resume` | 신규 주문 재개 |
| `POST /api/v1/bot/control/rulepack/apply` | 검증 통과 RulePack 적용 |
| `POST /api/v1/bot/control/rulepack/reject` | RulePack 적용 거부 |
| `POST /api/v1/bot/control/scheduler/run` | 특정 배치 작업 수동 실행 |
| `POST /api/v1/bot/control/reconnect` | WebSocket/KIS 연결 재시도 |

### 5.4 공통 응답 구조 권장안

```text
success
timestamp
request_id
data
warnings
block_reason
human_message
```

설명:

- `data`: 기계가 읽는 구조화 데이터
- `warnings`: 경고 목록
- `block_reason`: 차단 사유
- `human_message`: PM/운영자용 자연어 요약

### 5.5 PM이 알아야 할 API 방향

- 운영 화면은 가능하면 여러 API를 조합하지 말고, `overview` 같은 집계형 API를 우선 사용한다.
- 반면 분석/복기 화면은 세부 로그를 깊게 읽어야 하므로 세분화 API를 함께 둔다.

## 6. 데이터 저장 설계

아래는 우선 검토할 테이블 후보와 저장 이유다.  
실제 DB 종류는 추후 확정하되, 구조는 관계형 테이블 + 일부 JSON 컬럼 혼합이 적합하다.

### 6.1 핵심 운영 테이블

| 테이블 후보 | 저장 이유 |
|---|---|
| `rulepack_versions` | 일자별 RulePack 원본, 상태, 변경 이력 저장 |
| `rulepack_human_summaries` | 운영자 표시용 자연어 요약 저장 |
| `daily_universes` | 오늘 유니버스 결과와 소스 API 기록 |
| `universe_candidates` | 후보별 점수, 제외 사유, 최종 통과 여부 저장 |
| `live_candidate_snapshots` | 장중 시점별 판단 근거 저장 |
| `decision_events` | 매수/매도/대기/차단 판단 이벤트 저장 |
| `order_requests` | 주문 요청 데이터 저장 |
| `order_executions` | 주문 접수/체결/실패 결과 저장 |
| `order_blocks` | 주문 차단 사유 저장 |
| `positions` | 현재 보유 상태와 평균단가, 손익 상태 저장 |
| `risk_state_history` | 일중 손익률, 제한 상태 변화 이력 저장 |
| `scheduler_runs` | 배치 작업 시작/완료/실패 기록 저장 |
| `api_call_logs` | 외부 API 호출량, 실패, 제한 초과 상황 저장 |
| `system_health_logs` | 연결 상태, 장애, 복구 이력 저장 |

### 6.2 학습/복기 테이블

| 테이블 후보 | 저장 이유 |
|---|---|
| `learning_entry_snapshots` | 진입 시점의 조건 조합 저장 |
| `learning_skipped_candidates` | 미진입 후보와 이후 성과 저장 |
| `learning_blocked_candidates` | 차단 종목과 차단 적중 여부 저장 |
| `trade_outcomes` | 보유시간, 최대유리/불리변동, 청산 사유 저장 |
| `daily_review_reports` | 장마감 복기 리포트 저장 |
| `strategy_memory` | 좋은 전략/나쁜 전략 패턴 누적 저장 |
| `rulepack_improvement_suggestions` | 다음날 RulePack 개선 제안 저장 |

### 6.3 저장 설계 원칙

1. 운영용 현재 상태와 감사용 이력을 분리한다.
2. 숫자형 결과만 저장하지 말고 판단 근거도 저장한다.
3. 원본 응답 전문은 필요 최소한으로 저장하고, 민감 정보는 제외한다.
4. 학습 데이터는 나중에 통계 분석하기 쉽도록 구조화된 컬럼 중심으로 저장한다.

## 9. 테스트/검증 계획

본 문서는 구현이 아니라 계획 문서이므로, 아래는 개발 이후 반드시 따라야 할 검증 계획이다.

### 9.1 검증 원칙

- 구현 완료와 테스트 완료를 분리한다.

- 기존 기능 회귀를 반드시 확인한다.
- 장중 판단 로직은 숫자 결과뿐 아니라 차단 사유까지 검증한다.

### 9.2 최소 테스트 범주

1. **KIS Gateway 테스트**
   - 토큰 발급 성공/실패

   - 실전/모의 환경 분기
2. **RateLimiter 테스트**
   - 초당 호출량 제한
   - `EGW00201` 발생 시 감속
   - 신규 계정 3 TPS 정책 대응
3. **Universe Engine 테스트**
   - 랭킹 API 기반 후보 생성
   - API 실패 시 부분 실패 허용
   - 중복 종목 제거
4. **RulePack Engine 테스트**
   - JSON Schema 검증
   - 값 범위/금지 조건 검증
   - human summary 동시 저장 검증
5. **Decision Engine 테스트**
   - 진입 가능/대기/차단 상태 전이
   - 청산 조건 충족 판단
6. **Risk & Order Guard 테스트**
   - 손실한도 초과 차단

   - 데이터 지연/스프레드 과다 차단
7. **운영 API 테스트**
   - `/api/v1/bot/overview` 응답
   - 주문/차단/리스크 조회 응답
   - 제어 API 권한 및 안전성
8. **학습/복기 테스트**
   - 미진입 후보 후속 성과 저장
   - 일일 리뷰 생성
   - 다음날 개선안 생성

### 9.3 검증 방법

- 단위 테스트: 모듈별 규칙 검증
- 통합 테스트: 유니버스 생성 → 판단 → 차단/주문 → 저장

- 왜 주문했는지 설명 가능한가

## 10. 리스크와 보류 항목

### 10.1 주요 리스크

1. **KIS 호출 제한 리스크**
   - 신규 고객 3 TPS 정책으로 인해 초기 실전 속도가 크게 제한될 수 있다.
2. **모의투자 지원 범위 한계**
   - 일부 유니버스/랭킹 API는 모의 환경 지원이 제한적일 수 있다.
3. **실시간 데이터 지연 리스크**
   - WebSocket 지연 또는 끊김 시 잘못된 주문 판단 가능성이 있다.
4. **과도한 단건 조회 리스크**
   - 후보 축소 전 단건 조회를 많이 하면 API 한도와 성능이 동시에 악화된다.
5. **자동학습 오작동 리스크**
   - 장중 자가 수정은 금지해야 하며, 장마감 후 검증 반영 구조를 지켜야 한다.
6. **설명 불가능성 리스크**
   - 주문 근거와 차단 근거를 저장하지 않으면 운영 신뢰가 무너진다.

### 10.2 보류 항목

아래는 기획은 필요하지만 구현 우선순위는 뒤로 미루는 항목이다.

- 모델 기반 고급 학습
- 해외주식 확장
- 모바일 전용 제어 화면
- 다중 브로커 추상화
- 자동 전략 A/B 테스트

### 10.3 확인 필요 항목

- 실제 운영 DB 종류 확정
- 단타봇 API 허브의 최종 역할 범위 확정
- 모의투자 환경에서 실제 검증 가능한 API 목록 확정
- RulePack 승인 절차를 사람 승인으로 둘지 시스템 자동 승인으로 둘지 확정

## 11. 단타봇 API 허브와의 연결 전략

현재 문서들에는 "단타봇 API 허브" 상세 정의가 아직 없다.  
따라서 본 문서에서는 **허브를 운영 콘솔, 외부 자동화, 향후 관리자 도구가 함께 붙는 표준 연결 지점**으로 가정하고 전략을 제안한다.

### 11.1 허브의 권장 역할

- 여러 화면/도구가 같은 백엔드 상태를 읽는 공통 창구
- 단타봇 핵심 엔진을 직접 노출하지 않고, 표준 API만 공개하는 완충층
- 향후 알림 시스템, 리포트 시스템, 관리자 도구와 연결되는 관문

### 11.2 연결 방식 권장안

1. Python 엔진은 내부 서비스로 유지한다.
2. 운영 외부에는 `/api/v1/bot` 표준 API만 공개한다.
3. 단타봇 API 허브는 이 표준 API를 읽고 조합하는 방식으로 연결한다.
4. 주문 직접 호출은 허브가 하지 않고, 허브는 제어 API를 통해서만 간접 요청한다.

### 11.3 허브 연동 범위

우선 연결:

- `overview`
- `rulepack/today`
- `universe`
- `decisions/live`
- `orders`
- `blocks`
- `risk`
- `review/daily`

후순위 연결:

- `learning/summary`
- 세부 API 호출 로그
- 전략 메모리 상세 분석

### 11.4 허브 연결 시 장점

- 화면이 바뀌어도 엔진은 그대로 유지된다.
- 운영 콘솔, 리포트 도구, 알림 도구를 같은 데이터 기준으로 묶을 수 있다.
- 보안상 외부에서 엔진 내부 모듈을 직접 만지지 않게 할 수 있다.

### 11.5 허브 연결 시 주의점

- 허브가 엔진 로직을 다시 구현하면 안 된다.
- 허브는 "판단"보다 "조회/중계/제어 요청"에 집중해야 한다.
- 권한 없는 긴급정지/재개 호출을 막기 위한 인증 구조가 필요하다.

## 마무리 정리

Python 백엔드 v2의 핵심은 단순한 매매 스크립트가 아니라, 아래 5가지를 동시에 만족하는 운영형 시스템을 만드는 것이다.

1. KIS 호출 제한을 견디는 안정적 연동
2. 유니버스 API 우선 구조
3. RulePack 기반 자동 판단과 강한 주문 가드
4. 주문/차단/복기 근거를 남기는 감사 가능 구조
5. 장마감 학습 후 다음날 반영되는 안전한 개선 루프

즉, 이 백엔드는 "매매를 하는 프로그램"을 넘어서, **설명 가능하고 차단 가능하며 개선 가능한 자동매매 운영 시스템**으로 설계되어야 한다.
