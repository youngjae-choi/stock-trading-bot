# AI 자동매매 단타봇 운영 콘솔 화면기획서 v1.0

## 1. 문서 개요

이 문서는 한국주식 KIS Open API 기반 **AI 하이브리드 자동매매 단타봇**의 운영 콘솔 화면기획서입니다.

본 시스템은 사용자가 직접 종목을 선택하고 주문을 승인하는 매매 도구가 아니라, **AI가 생성한 RulePack을 시스템이 자동 실행하고, 사용자는 전체 자동매매 사이클과 리스크 상태를 감시하는 관제형 시스템**입니다.

---

### 1.1 문서 목적

본 기획서의 목적은 다음과 같습니다.

| **항목** | **내용** |
|---|---|
| 화면 목적 정의 | 자동매매 시스템에서 사용자가 무엇을 봐야 하는지 정의 |
| 정보구조 설계 | 메뉴, 화면, 데이터 흐름 정리 |
| 사용자 역할 정립 | 매매 판단자 아닌 시스템 운영자로 정의 |
| AI/시스템 역할 분리 | AI는 룰 생성, 시스템은 정량 계산과 실행 |
| HTML 프로토타입 기준 | 추후 제작할 반응형 HTML 화면의 기준 제공 |
| PC/모바일 대응 | PC 웹 우선, 모바일 웹 확인용 설계 |

---

### 1.2 핵심 설계 방향

이 시스템의 화면은 다음 질문에 답해야 합니다.

| **시점** | **화면이 답해야 하는 질문** |
|---|---|
| 장 시작 전 | 오늘 AI가 어떤 조건으로 운용할 예정인가? |
| 스크리닝 후 | 2,500개 종목이 어떤 기준으로 200개, 15개로 줄었는가? |
| 장중 | 어떤 종목이 매수조건 또는 매도조건을 기다리고 있는가? |
| 주문 발생 시 | 왜 이 주문이 실행되었는가? |
| 주문 미발생 시 | 왜 아직 주문하지 않았는가? |
| 주문 차단 시 | 어떤 조건 때문에 주문이 차단되었는가? |
| 장마감 후 | 시스템은 오늘 룰대로 움직였는가? |

## 2. 구현 플랫폼 판단

현재 단계에서는 **PC 웹 기반 운영 콘솔**이 가장 적합합니다.

---

### 2.1 권장 구현 방식

| **구분** | **권장 방식** | **이유** |
|---|---|---|
| 자동매매 엔진 | Python 백엔드 서비스 | KIS API, 룰 엔진, 스케줄러, 주문 실행에 적합 |
| 운영 콘솔 | PC 웹 / 반응형 웹 | 원격 확인, 유지보수, 배포가 쉬움 |
| 모바일 확인 | 모바일 웹 / 추후 PWA | 장중 상태 확인, 알림 확인용 |
| PC 앱 | 후순위 | 별도 배포/업데이트 부담이 큼 |
| 모바일 네이티브 앱 | 후순위 | 푸시/백그라운드 필요 시 검토 |

---

### 2.2 웹이 적합한 이유

자동매매 시스템에서 중요한 것은 UI 성능보다 **엔진의 안정성**입니다.

따라서 구조는 다음처럼 분리하는 것이 좋습니다.

```text
Python 자동매매 엔진
- KIS API 연결
- AI RulePack 반영
- 룰 엔진 실행
- 주문/청산/리스크 처리
- 로그 저장

웹 운영 콘솔
- 현재 상태 확인
- RulePack 확인
- Funnel 진행 확인
- Live Decision 확인
- 주문/차단/리스크 로그 확인
- 복기 리포트 확인
```

웹 화면이 꺼져도 자동매매 엔진은 계속 돌아야 합니다.

즉, 화면은 **조종석이 아니라 관제실**입니다.  
조종은 시스템이 하고, 사용자는 상태를 감시합니다.

## 3. 사용자 역할 정의

사용자는 실시간 매매 판단자가 아닙니다.

---

### 3.1 사용자 역할

| **역할** | **설명** |
|---|---|
| 운영자 | 자동매매 시스템이 정상 작동하는지 감시 |
| 관리자 | RulePack, 리스크 한도, API 상태 확인 |
| 감사자 | 장마감 후 거래 근거와 룰 준수 여부 확인 |
| 예외 처리자 | 긴급정지, 시스템 중단, 데이터 오류 대응 |

---

### 3.2 사용자가 하지 않는 것

| **항목** | **이유** |
|---|---|
| 매번 종목 직접 선택 | AI/시스템의 자동 스크리닝 목적과 충돌 |
| 매수 타이밍 수동 승인 | 자동매매의 의미가 약해짐 |
| 매도 타이밍 수동 승인 | 손절/익절/트레일링은 기계적으로 실행되어야 함 |
| 전체 유니버스 수동 검토 | 200개 종목 전체를 사람이 볼 필요 없음 |

---

### 3.3 사용자가 확인해야 하는 것

| **항목** | **설명** |
|---|---|
| 오늘 AI RulePack | 오늘 어떤 룰로 움직이는지 확인 |
| Funnel 결과 | 2,500개 → 200개 → 15개 축소 과정 확인 |
| 매수조건 대기 | 어떤 종목이 어떤 조건을 기다리는지 확인 |
| 매도조건 대기 | 보유 종목이 어떤 청산 조건을 기다리는지 확인 |
| 주문 실행/차단 | 왜 주문했는지, 왜 차단했는지 확인 |
| 리스크 상태 | 일일 손실 한도, 주문 제한 상태 확인 |
| 데이터/API 상태 | KIS, DART, 뉴스, AI 검색 상태 확인 |
| 복기 결과 | 룰 준수율, 미진입 사유, 개선 제안 확인 |

## 4. AI와 시스템 역할 분리

이 시스템의 핵심은 **AI는 해석과 룰 생성**, **시스템은 정량 계산과 실행**을 담당하는 것입니다.

---

### 4.1 AI 담당 영역

| **영역** | **역할** |
|---|---|
| 시장 톤 분석 | 코스피/코스닥 분위기, 리스크 수준 판단 |
| 외부 뉴스 분석 | 테마, 섹터, 종목 뉴스, 공시 해석 |
| 정성 데이터 판단 | 시장 분위기, 테마 강도, 섹터 로테이션 |
| RulePack 생성 | 오늘의 유니버스 조건, 후보 스크리닝 가중치, 진입/청산 룰 생성 |
| 복기 리포트 | 장마감 후 거래 결과 분석 및 개선 제안 |

---

### 4.2 시스템 담당 영역

| **영역** | **역할** |
|---|---|
| 정량 데이터 수집 | 현재가, 호가, 거래량, 거래대금, 체결, 잔고 |
| 재무 데이터 수집 | PER, PBR, ROE, 부채비율 등 |
| 기술 지표 계산 | VWAP, RSI, 이동평균, 거래량 배율 |
| Funnel 실행 | 전체 종목 → 유니버스 → 후보군 |
| 룰 엔진 실행 | AI RulePack 조건을 실시간으로 평가 |
| 주문 실행 | KIS 주문 API 호출 |
| 리스크 차단 | 손실 한도, 포지션 한도, API 오류 시 주문 차단 |
| 로그 저장 | 판단, 주문, 체결, 차단, 오류 기록 |

---

### 4.3 전체 데이터 흐름

```text
외부 뉴스 / 테마 / 시장 정보
        ↓
AI 시장 톤 분석
        ↓
AI RulePack 생성
        ↓
시스템 정량 데이터 수집
        ↓
Layer 1 Universe 필터
        ↓
Layer 2 후보 스크리닝
        ↓
Layer 3 실시간 진입 트리거 감시
        ↓
자동 주문 / 자동 청산
        ↓
로그 저장 / 장마감 복기
```

## 7.2 AI RulePack

AI RulePack 화면은 AI가 오늘 시스템에 입력한 운용 조건을 확인하는 화면입니다.

---

### 화면 목적

- AI 시장 톤 분석 결과 확인
- 오늘의 유니버스 조건 확인
- 후보 스크리닝 가중치 확인
- 실시간 매수/매도 트리거 확인
- RulePack JSON 확인
- AI 정성 분석과 시스템 정량 조건의 연결 확인

---

### 주요 표시 정보

| **영역** | **표시 항목** |
|---|---|
| RulePack 요약 | ID, 생성시각, 적용상태, 유효기간 |
| 시장 톤 | KOSPI, KOSDAQ, 리스크 레벨, AI 코멘트 |
| 외부 분석 | 뉴스, 테마, 섹터, 미국장 영향 |
| Layer 1 조건 | 시총, 거래대금, 상장일, 관리/경고 제외 |
| Layer 2 조건 | 기술/기초/테마 가중치 |

## 7.3 Funnel Monitor

Funnel Monitor는 전체 종목이 어떤 기준으로 줄어들었는지 보여주는 화면입니다.

---

### 화면 목적

- Layer 1 필터 결과 확인
- Layer 2 후보 스크리닝 결과 확인
- 통과/탈락 사유 확인
- 후보 15개 선정 근거 확인
- AI 정성 점수와 시스템 정량 점수 결합 결과 확인

---

### Funnel 단계

```text
전체 종목 2,500
  ↓
Layer 1 Universe 200
  ↓
Layer 2 후보 15
  ↓
Layer 3 매수조건 대기
  ↓
실체결
```

---

### 주요 표시 정보

| **영역** | **표시 항목** |
|---|---|
| Funnel 요약 | 전체, Layer 1 통과, Layer 2 통과, 매수대기, 체결 |
| Layer 1 탈락 사유 | 시총 미달, 거래대금 미달, 상장일 미달, 관리/경고 |
| Layer 2 점수 | 기술점수, 기초점수, 테마점수, 총점 |
| 후보 15개 | 종목코드, 종목명, 총점, AI 신뢰도, 선정 사유 |
| 이상 징후 | 후보 수 과다/과소, 데이터 누락, 점수 이상치 |

---

### 후보 15개 테이블 컬럼

| **컬럼** | **설명** |
|---|---|
| 종목코드 | 6자리 문자열 |
| 종목명 | 종목명 |
| 기술점수 | 갭, 거래량, 신고가 등 |
| 기초점수 | PER, ROE, 부채비율 |
| 테마점수 | 뉴스/테마/수급 |
| 총점 | 가중합 점수 |
| AI 신뢰도 | 정성 분석 신뢰도 |
| 현재 상태 | 감시중, 매수대기, 제외 |
| 선정 사유 | 후보군 포함 근거 |

## 14. 최종 결론

본 시스템의 UI는 일반 트레이딩 화면이 아니라 **자동매매 의사결정 관제센터**입니다.

따라서 화면의 중심은 다음이 아닙니다.

```text
종목 검색
현재가 확인
수동 매수
수동 매도
```

화면의 중심은 다음이어야 합니다.

```text
오늘 AI가 만든 RulePack
2,500 → 200 → 15 Funnel 결과
현재 매수조건 대기 상태
현재 매도조건 대기 상태
주문 실행/차단 사유
KIS/API 데이터 품질
리스크 한도
장마감 감사 로그
```

이 기획서 v1.0을 기준으로 HTML은 **PC 우선 반응형 웹 콘솔**로 제작합니다.  
디자인은 과한 AI 대시보드 느낌을 제거하고, 실제 사람이 만든 금융/운영 시스템처럼 **텍스트 중심, 고밀도 정보, 다크/라이트 테마, 명확한 상태 색상**을 기준으로 합니다.
