맞습니다. 이건 **RulePack 설계에서 아주 중요한 원칙**입니다.  
정확히 말하면 RulePack은 **시스템 내부에서는 기계가 읽는 구조화 스크립트/JSON으로 주고받고**, 사용자가 보는 화면에서는 **자연어 요약과 판단 근거 중심으로 보여줘야 합니다.**

즉, 사용자는 이런 JSON을 매번 읽으면 안 됩니다.

```json
{
  "min_volume_ratio": 2.0,
  "rsi_min": 40,
  "rsi_max": 70,
  "max_spread_rate": 0.003
}
```

대신 화면에서는 이렇게 보여줘야 합니다.

```text
오늘은 거래량이 평소보다 2배 이상 증가하고,
VWAP 위에서 유지되는 종목만 진입 대상으로 봅니다.
RSI가 40~70 사이일 때만 진입하며,
호가 스프레드가 0.3%를 넘으면 주문하지 않습니다.
```

이렇게 해야 시스템은 정확하게 실행하고, 사용자는 쉽게 이해할 수 있습니다.

## 1. RulePack은 이중 구조로 설계해야 합니다

RulePack은 하나의 개념이지만, 실제로는 **두 가지 표현 방식**을 가져야 합니다.

| **구분** | **대상** | **형식** | **목적** |
|---|---|---|---|
| Machine RulePack | 시스템/룰 엔진/API | JSON 또는 DSL Script | 정확한 실행 |
| Human Rule Summary | 사용자/운영자 | 자연어 설명 | 이해와 감시 |
| Audit Explanation | 복기/감사 | 자연어 + 근거 데이터 | 사후 검증 |
| AI Reasoning Note | AI/리포트 | 자연어 분석 | 전략 의도 설명 |

핵심은 이겁니다.

```text
시스템끼리 주고받는 내용 = 구조화된 RulePack
사용자에게 보여주는 내용 = 자연어 운영 설명
```

---

## 2. RulePack 데이터 구조 권장안

RulePack은 내부적으로 아래처럼 나누는 것이 좋습니다.

```text
RulePack
├─ metadata
├─ machine_rules
├─ human_summary
├─ ai_rationale
├─ risk_policy
├─ audit_policy
└─ version_info
```

### 구조 역할

| **영역** | **역할** |
|---|---|
| `metadata` | RulePack ID, 날짜, 생성시각, 적용상태 |
| `machine_rules` | 룰 엔진이 직접 실행할 조건 |
| `human_summary` | 사용자 화면에 보여줄 자연어 설명 |
| `ai_rationale` | AI가 왜 이 조건을 만들었는지 설명 |
| `risk_policy` | 리스크 제한 규칙 |
| `audit_policy` | 복기/학습을 위해 저장할 항목 |
| `version_info` | 변경 이력, 이전 RulePack과 차이 |

## 4. 사용자에게 보여줄 자연어 RulePack 예시

같은 RulePack을 사용자 화면에서는 아래처럼 보여줘야 합니다.

---

### 오늘의 운용 요약

```text
오늘 시스템은 코스닥 중심의 단타 기회를 우선 감시합니다.
시장 리스크는 중간 수준으로 판단되어 자동매매는 가능하지만,
종목당 최대 비중은 10%로 제한합니다.

전체 종목 중 시가총액 500억 원 이상,
20일 평균 거래대금 10억 원 이상,
상장 60일 이상인 종목만 1차 유니버스에 포함합니다.
관리종목과 투자경고 종목은 자동 제외합니다.
```

---

### 후보 선정 기준

```text
1차 필터를 통과한 종목 중에서 기술적 조건을 가장 중요하게 봅니다.

오늘의 후보 점수는 다음 비중으로 계산됩니다.

- 기술적 조건: 60%
- 기초적 조건: 25%
- 테마/시장 조건: 15%

기술적 조건에서는 갭상승, 거래량 급증, 신고가 여부를 봅니다.
기초 조건에서는 PER, ROE, 부채비율을 확인합니다.
테마 조건에서는 뉴스, 외국인·기관 수급, 시장 테마 강도를 반영합니다.
```

---

### 매수 조건

```text
시스템은 후보 종목 15개를 실시간으로 감시합니다.

아래 조건을 모두 충족할 때만 매수 주문을 검토합니다.

- 현재가가 VWAP 위에 있어야 합니다.
- 거래량이 평소보다 2배 이상 증가해야 합니다.
- 현재가가 20일 이동평균선 위에 있어야 합니다.
- RSI는 40 이상 70 이하 범위여야 합니다.
- 호가 스프레드는 0.3% 이하여야 합니다.
- 시장 지수 흐름과 종목 움직임이 크게 어긋나지 않아야 합니다.
- AI 신뢰도는 0.65 이상이어야 합니다.
```

---

### 매도 조건

```text
매수 후에는 시스템이 자동으로 청산 조건을 감시합니다.

- 손실이 -1.5%에 도달하면 즉시 손절합니다.
- 수익이 +3.0%에 도달하면 익절합니다.
- 수익이 +2.0% 이상 발생한 후에는 트레일링 스탑을 활성화합니다.
- 고점 대비 1.0% 하락하면 트레일링 청산합니다.
- 진입 후 30분 동안 +0.5% 이상 수익이 나지 않으면 시간손절을 검토합니다.
```

---

### 리스크 제한

```text
오늘의 리스크 제한은 다음과 같습니다.

- 당일 손실이 -2.0%에 도달하면 신규 매수를 전면 중단합니다.
- 주간 손실이 -5.0%에 도달하면 전략 재검토 상태로 전환합니다.
- 월간 손실이 -8.0%에 도달하면 시스템을 일시 정지합니다.
- 동시에 보유할 수 있는 종목은 최대 5개입니다.
- 한 종목에 투입할 수 있는 최대 비중은 총자산의 10%입니다.
```

이게 사용자가 실제로 봐야 하는 RulePack입니다.

## 8. RulePack 생성 파이프라인도 바뀌어야 합니다

AI가 RulePack을 생성할 때는 **기계용 룰과 사용자용 설명을 같이 생성**해야 합니다.

```text
AI 시장 분석
   ↓
RulePack 초안 생성
   ↓
machine_rules 생성
   ↓
human_summary 생성
   ↓
시스템 JSON Schema 검증
   ↓
리스크 제한 검증
   ↓
Rule Engine 적용
   ↓
사용자 화면에는 human_summary 표시
```

---

### AI 출력 형식 권장안

AI에게 RulePack을 만들게 할 때는 이런 구조를 요구해야 합니다.

```json
{
  "machine_rules": {
    "layer1_universe": {},
    "layer2_screening": {},
    "layer3_entry": {},
    "exit_rules": {},
    "risk_limits": {}
  },
  "human_summary": {
    "operation_summary": "",
    "market_tone_summary": "",
    "universe_summary": "",
    "screening_summary": "",
    "entry_summary": "",
    "exit_summary": "",
    "risk_summary": "",
    "changed_rules": []
  },
  "ai_rationale": {
    "market_reasoning": "",
    "theme_reasoning": "",
    "risk_reasoning": ""
  }
}
```

다만 중요한 점은, **AI가 만든 machine_rules를 그대로 믿으면 안 됩니다.**  
반드시 시스템 검증기를 통과해야 합니다.

## 9. 시스템 검증기는 자연어가 아니라 machine_rules만 검증합니다

검증 대상은 자연어가 아니라 내부 룰입니다.

| **검증 항목** | **대상** |
|---|---|
| JSON Schema 검증 | `machine_rules` |
| 값 범위 검증 | `rsi_min`, `rsi_max`, `stop_loss_rate` 등 |
| 리스크 제한 검증 | 손실한도, 포지션 한도 |
| 금지 조건 검증 | AI가 손절 완화, 손실한도 확대 시도 여부 |
| 실행 가능성 검증 | 룰 엔진이 해석 가능한 구조인지 |

예:

```text
AI가 자연어로 “오늘은 조금 공격적으로 운용합니다”라고 써도,
machine_rules에서 일일 손실한도를 -2%에서 -4%로 바꾸면 시스템이 차단해야 합니다.
```

## 10. RulePack 화면에서 보여줄 “변경사항”이 중요합니다

사용자는 매일 전체 룰을 다 읽기보다 **어제와 뭐가 달라졌는지**를 봐야 합니다.

### 변경사항 예시

```text
어제 대비 변경된 조건

1. 거래량 기준 강화
   - 기존: 1.8배 이상
   - 오늘: 2.0배 이상
   - 이유: 전일 거래량 급증 종목의 추세 지속률이 낮았음

2. RSI 상단 기준 하향
   - 기존: 72 이하
   - 오늘: 70 이하
   - 이유: RSI 70 이상 진입의 시간손절 비율이 증가

3. 테마 가중치 축소
   - 기존: 20%
   - 오늘: 15%
   - 이유: 뉴스 반응보다 실제 거래대금 반응을 우선하기 위함
```

이 영역은 AI RulePack 화면 상단에 있어야 합니다.

## 11. 복기/자동학습에서도 같은 원칙이 적용됩니다

복기에서도 내부 학습 데이터는 수치/구조화 데이터로 저장하고, 사용자에게는 자연어로 보여줘야 합니다.

### 내부 학습 데이터

```json
{
  "pattern_type": "BAD_ENTRY",
  "condition": {
    "rsi_range": "68-70",
    "volume_ratio": "2.0-2.5",
    "spread_rate": ">0.25"
  },
  "sample_count": 18,
  "win_rate": 0.33,
  "avg_return": -0.0021,
  "recommendation": {
    "target_rule": "rsi_max",
    "suggested_value": 67
  }
}
```

### 사용자 표시

```text
최근 데이터에서는 RSI가 68~70 구간이고,
```

```md
## RulePack 표현 원칙

RulePack은 내부 실행용 표현과 사용자 표시용 표현을 분리한다.

### 1. 내부 실행용 RulePack
- 시스템, 룰 엔진, API, 검증기가 사용하는 구조화 데이터
- JSON 또는 DSL Script 형식
- 정확한 수치 조건과 상태값 포함
- 자동매매 실행의 기준이 됨

### 2. 사용자 표시용 RulePack
- 운영자가 이해할 수 있는 자연어 설명
- 오늘 운용 요약, 매수 조건, 매도 조건, 리스크 제한, 변경사항 포함
- 원본 JSON은 고급 보기로 제공
- 사용자는 JSON이 아니라 자연어 요약을 기본으로 확인

### 3. 검증 원칙
- 시스템 검증은 자연어가 아니라 machine_rules를 대상으로 한다.
- AI가 생성한 RulePack은 JSON Schema, 리스크 제한, 실행 가능성 검증을 통과해야 한다.
- 손절폭 완화, 손실한도 확대, 포지션 한도 증가 등 위험한 변경은 자동 차단한다.

### 4. 복기 원칙
- 내부 학습 데이터는 구조화된 수치 데이터로 저장한다.
- 사용자에게는 자연어 복기 리포트로 보여준다.
- 좋은 전략, 나쁜 전략, 좋은 타이밍, 차단 사유는 모두 학습 메모리에 저장한다.
```

## 13. 최종 정리

정확한 방향은 이겁니다.

```text
AI ↔ 시스템:
JSON / DSL / 구조화 RulePack

시스템 ↔ 룰 엔진:
검증된 machine_rules

시스템 ↔ 사용자:
자연어 요약 / 상태 설명 / 변경사항 / 차단 사유

복기 시스템 ↔ 학습 메모리:
구조화 데이터

복기 시스템 ↔ 사용자:
자연어 리포트
```

따라서 화면에는 **원본 JSON을 메인으로 보여주면 안 됩니다.**  
원본 JSON은 “개발자 보기” 또는 “원본 RulePack 보기”에 넣고, 기본 화면은 자연어 카드 중심이어야 합니다.

이 원칙을 반영하면 자동매매 콘솔은 사람이 이해하기 쉬우면서도, 내부적으로는 정확하고 재현 가능한 시스템이 됩니다.
