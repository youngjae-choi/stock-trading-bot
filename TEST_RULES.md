# TEST_RULES.md

## 목적

이 문서는 기능 개발, 버그 수정, 리팩토링, UI 변경 이후  
**무엇을 어떻게 검증할지**에 대한 공통 기준을 정의한다.

목표는 다음과 같다.

- "코드는 바뀌었지만 검증이 부족한 상태"를 방지한다
- AI 에이전트가 구현만 하고 테스트를 생략하지 않도록 한다
- 최소한의 수동 검증과 가능한 자동 검증을 습관화한다
- 기능 단위 / 예외 상황 / 회귀 위험까지 함께 점검한다
- 완료 보고에 테스트 결과가 포함되도록 강제한다

이 문서는 개별 기능 명세보다 우선하지 않는다.  
다만 별도 지시가 없으면 본 문서의 기준을 기본값으로 따른다.

## 핵심 원칙

### 1. 구현 완료와 테스트 완료는 다르다

기능이 "만들어졌다"는 것과  
기능이 "검증되었다"는 것은 다르다.

다음 상태는 완료로 보지 않는다.

- 코드만 작성함
- 화면만 얼핏 확인함
- 정상 케이스만 한 번 실행함
- 에러 케이스를 전혀 확인하지 않음
- 기존 기능이 깨졌는지 확인하지 않음

완료는 최소한 아래를 포함한다.

- 핵심 동작 확인
- 대표 예외 상황 확인
- 회귀 위험 확인
- 결과 보고

---

### 2. 테스트는 정상 흐름만이 아니라 실패 흐름까지 포함한다

좋은 테스트는 "될 때"만 보는 것이 아니다.

최소한 아래를 나누어 검토한다.

- 정상 입력
- 잘못된 입력
- 데이터 없음
- 로딩 중
- 실패 응답
- 중복 실행
- 경계값
- 권한 문제
- timeout 또는 지연
- 부분 성공

---

### 3. 자동 테스트가 없더라도 수동 테스트는 반드시 한다

자동 테스트가 없는 프로젝트라도 검증이 면제되는 것은 아니다.

원칙:
- 자동 테스트 가능하면 추가
- 자동 테스트가 어려우면 수동 테스트 명시
- 최소한 재현 절차와 기대 결과를 남김
- "테스트는 못했음"으로 끝내지 않음

---

### 4. 테스트는 기능 단위뿐 아니라 사용자 흐름 단위로 본다

버튼 하나만 눌러보는 것으로 충분하지 않을 수 있다.

검토 예:
- 생성 → 조회 → 수정 → 삭제
- 검색 → 결과 없음 → 필터 초기화
- 저장 실패 → 수정 유지 → 재시도
- 로그인 만료 → 재로그인 → 다시 시도

---

### 5. 회귀 테스트를 항상 고려한다

새 기능을 넣었더라도 기존 기능이 깨지면 실패다.

반드시 검토:
- 기존 버튼/기존 API/기존 화면 영향
- 공용 컴포넌트 변경 영향
- 상태 관리 변경 영향
- 스타일 변경에 따른 다른 화면 깨짐
- 기존 데이터 구조와의 호환성

## 테스트 수준 정의

테스트는 아래 수준으로 나누어 생각한다.

### 1. Smoke Test
가장 기본적인 생존 확인

목적:
- 앱이 열리는가
- 화면이 깨지지 않는가
- 버튼이 눌리는가
- 치명적 에러가 없는가

예:
- 페이지 진입 가능
- 기본 버튼 클릭 가능
- 콘솔 에러 폭발 없음

---

### 2. Functional Test
기능 요구사항 충족 확인

목적:
- 요구한 기능이 실제로 동작하는지 확인

예:
- 저장 버튼 클릭 시 저장된다
- 검색 시 결과가 바뀐다
- 삭제 확인 모달이 뜬다

---

### 3. State Test
상태별 동작 확인

목적:
- initial / loading / empty / success / error 상태가 자연스러운지 확인

예:
- 로딩 중 스피너 표시
- 데이터 없음 상태 문구 표시
- 실패 시 재시도 버튼 표시

---

### 4. Edge Case Test
경계값 / 예외 상황 확인

목적:
- 입력 이상치, 중복 클릭, null 데이터, 긴 텍스트 등 확인

예:
- 필수값 누락
- 같은 버튼 연속 클릭
- 응답 필드 누락
- 매우 긴 이름 입력

---

### 5. Regression Test
기존 기능 영향 확인

목적:
- 이번 변경으로 다른 기능이 망가지지 않았는지 확인

예:
- 공용 버튼 스타일 수정 후 기존 페이지 확인
- API 응답 처리 공통부 수정 후 다른 화면 확인

---

## 기본 테스트 분류

기능 구현 시 최소 아래 항목을 점검한다.

### A. 정상 흐름 테스트

반드시 확인:
- 사용자가 의도한 대표 시나리오가 끝까지 동작하는가
- 성공 후 UI 반영이 보이는가
- 상태값이 일관되게 유지되는가

예:
- 폼 입력 후 저장 성공
- 항목 생성 후 목록 반영
- 검색어 입력 후 결과 표시

---

### B. 입력 검증 테스트

반드시 확인:
- 필수값 누락 시 오류 표시
- 잘못된 형식 시 오류 표시
- 길이 제한/범위 제한 동작
- 에러 메시지가 이해 가능한지

예:
- 이메일 형식 오류
- 숫자 범위 초과
- 너무 긴 텍스트 입력

---

### C. 로딩 테스트

반드시 확인:
- 로딩 상태가 보이는가
- 중복 클릭이 막히는가
- 기존 데이터가 있을 때 화면이 갑자기 비지 않는가
- 긴 요청에서 멈춘 것처럼 보이지 않는가

---

### D. 빈 상태 테스트

반드시 확인:
- 데이터 0건일 때 empty state가 보이는가
- 검색 결과 없음과 전체 빈 상태가 구분되는가
- 다음 행동 버튼이 필요한가

---

### E. 실패 테스트

반드시 확인:
- 네트워크 실패 시 사용자 메시지 표시
- 서버 실패 시 재시도 가능
- 입력값 유지 여부
- 부분 성공 시 구분 표시

---

### F. 권한/조건 테스트

필요한 경우 확인:
- 권한 없는 사용자는 버튼이 숨겨지거나 차단되는가
- 로그인 만료 시 적절히 유도되는가
- 상태 조건 미충족 시 비활성화 이유가 있는가

---

### G. 중복 실행 테스트

반드시 확인:
- 버튼 연타 시 중복 요청이 발생하지 않는가
- 저장 중 다시 저장되지 않는가
- 삭제 중 중복 삭제 요청이 막히는가

---

### H. 회귀 테스트

변경 영향이 있는 경우 확인:
- 기존 화면이 깨지지 않는가
- 공용 컴포넌트 사용처가 이상 없는가
- 다른 플로우에서 오류가 늘어나지 않는가

---

## UI 테스트 규칙

UI 관련 작업은 최소 아래를 확인한다.

### 1. 화면 진입
- 에러 없이 열린다
- 레이아웃이 무너지지 않는다
- 기본 텍스트와 버튼이 보인다

### 2. 버튼 동작
- 클릭 가능
- disabled 상태 적절
- 중복 클릭 방지
- 처리 중 시각적 상태 변화 존재

### 3. 폼 동작
- 입력 가능
- 검증 메시지 표시
- 제출 성공/실패 처리
- 실패 시 값 유지

### 4. 모달 동작
- 열림/닫힘 자연스러움
- ESC/취소/닫기 동작 확인
- 확인 버튼과 취소 버튼이 명확함
- 처리 중 의도치 않은 닫힘 여부 검토

### 5. 토스트/알림
- 성공/실패 시 적절한 메시지
- 중복 과다 노출 없음
- 토스트만 뜨고 실제 화면 반영이 빠지지 않음

### 6. 반응형
- 모바일 폭에서 깨지지 않음
- 버튼이 눌릴 정도 크기 유지
- 테이블/폼/모달이 화면 밖으로 과도하게 넘치지 않음

### 7. 접근성 최소 점검
- 키보드로 핵심 조작 가능
- 포커스가 보임
- 아이콘 버튼에 이름 있음
- 색상만으로 상태를 구분하지 않음

---

## API / 데이터 테스트 규칙

데이터 연동 기능은 아래를 기본 검토한다.

### 1. 정상 응답
- 기대한 데이터가 정상 표시되는가

### 2. 빈 응답
- 빈 배열 / null / undefined에 안전한가

### 3. 누락 필드
- 일부 필드가 없을 때도 화면이 깨지지 않는가

### 4. 에러 응답
- 4xx / 5xx / timeout 구분이 필요한가

### 5. 지연 응답
- 로딩 중 상태 유지가 자연스러운가

### 6. 중복 요청
- 동일 요청이 불필요하게 여러 번 나가지 않는가

### 7. 부분 성공
- 일부만 성공한 경우 전체 성공처럼 보이지 않는가

---

## 버그 수정 테스트 규칙

버그 수정은 단순히 "증상이 안 보인다"로 끝내지 않는다.

반드시 확인:
1. 원래 버그가 재현되었는가
2. 수정 후 재현이 안 되는가
3. 관련 인접 기능이 안 깨졌는가
4. 비슷한 조건에서도 같은 문제가 안 나는가

버그 수정 보고에는 최소 아래를 포함한다.

- 원래 증상
- 재현 조건
- 수정 방식
- 수정 후 검증 결과
- 회귀 영향 확인 범위

---

## 리팩토링 테스트 규칙

리팩토링은 외형상 변화가 없더라도 검증이 필요하다.

반드시 확인:
- 기능 결과 동일
- 주요 사용자 흐름 동일
- 에러 처리 동일 또는 개선
- 성능 저하 없음
- 테스트 기존 통과 여부 확인

---

## 테스트 우선순위 규칙

시간이 부족할 때도 우선순위는 지킨다.

### 최우선
- 치명적 에러
- 저장/삭제/결제 등 핵심 액션
- 데이터 유실 가능성
- 중복 실행 위험
- 권한 문제

### 다음
- empty / error / loading 상태
- 모바일 기본 확인
- 회귀 확인

### 그 다음
- 미세한 UI 디테일
- 추가 편의 기능
- 문구 미세 조정

---

## 최소 수동 테스트 세트

별도 지시가 없으면 기능 하나당 최소 아래를 수행한다.

### 공통 최소 세트
1. 정상 동작 1회
2. 실패 동작 1회
3. 빈 상태 또는 경계값 1회
4. 중복 클릭 또는 재시도 1회
5. 관련 기존 기능 1회 재확인

예:
- 저장 성공
- 저장 실패
- 필수값 누락
- 저장 버튼 연타
- 목록 갱신 확인

---

## 자동 테스트 작성 기준

가능하면 자동 테스트를 고려한다.

자동 테스트 후보:
- 순수 함수 로직
- 검증 로직
- 상태 전환 로직
- 공용 유틸
- 핵심 컴포넌트 동작
- API 응답별 분기 처리

자동 테스트를 우선 고려해야 하는 경우:
- 재발 가능성이 높은 버그
- 공용 로직
- 복잡한 조건 분기
- 사람이 매번 확인하기 번거로운 로직
- 회귀 영향이 큰 기능

자동 테스트가 어려운 경우:
- 애니메이션 미세 표현
- 브라우저/플랫폼 특화 UI 감각
- 외부 시스템 의존성이 큰 통합 흐름

이 경우에도 최소 수동 테스트는 반드시 남긴다.

---

## 테스트 데이터 규칙

테스트 시 아래 데이터를 고려한다.

- 정상 데이터
- 빈 데이터
- 최소 길이 데이터
- 최대 길이 데이터
- 특수문자 포함 데이터
- 한글/영문/숫자 혼합
- null 또는 누락 필드
- 중복 데이터
- 오래된 데이터/예상 밖 상태 데이터

---

## 모바일 / 반응형 테스트 규칙

UI가 있으면 최소 반응형 확인을 한다.

기본 확인:
- 작은 폭에서 버튼 클릭 가능
- 텍스트가 과하게 잘리지 않음
- 입력창 사용 가능
- 모달이 화면을 벗어나지 않음
- 테이블이 읽을 수 없는 상태가 되지 않음

권장 폭 예:
- 작은 모바일 폭
- 일반 모바일 폭
- 노트북 폭

정확한 수치는 프로젝트 기준이 있으면 그 기준을 따른다.

---

## 로그 / 디버깅 확인 규칙

실패 테스트 시 아래를 함께 확인한다.

- 사용자 메시지가 적절한가
- 로그가 남는가
- 로그에 민감정보가 없는가
- 실패 지점 추적이 가능한가
- console 에러가 과도하게 발생하지 않는가

---

## 금지사항

다음은 특별한 이유가 없는 한 금지한다.

- 테스트 없이 "완료" 보고
- 정상 케이스만 확인하고 끝내기
- 실패 시나리오 미검증
- 수동 테스트도 안 하고 자동 테스트도 없음
- 회귀 영향 무시
- 입력값 유실 여부 미확인
- 중복 클릭 미확인
- 모바일 확인 생략
- "로컬에서 잘 됨"만 적고 절차 없음
- 테스트 결과를 추상적으로만 보고

## AI 에이전트 작업 지침

AI 에이전트는 기능 구현 후 다음을 따른다.

1. 최소 수동 테스트 세트를 수행하거나, 수행 가능한 형태로 제시한다
2. 정상 / 실패 / 빈 상태 / 중복 실행을 기본 검토한다
3. 기존 기능 영향 여부를 확인한다
4. 가능하면 자동 테스트 추가 여부를 판단한다
5. 테스트하지 못한 항목이 있다면 숨기지 말고 명시한다
6. "어떻게 검증했는지"를 결과 보고에 포함한다

---

## 테스트 결과 보고 형식

구현 완료 후 보고에는 최소 아래를 포함한다.

### 1. 변경 파일
- 어떤 파일을 수정했는지

### 2. 검증한 항목
- 정상 동작
- 실패 동작
- empty 상태
- 로딩 상태
- 중복 클릭
- 회귀 확인

### 3. 테스트 방법
- 수동 테스트 절차
- 자동 테스트 명령어
- mock 또는 시뮬레이션 여부

### 4. 결과
- 통과
- 부분 통과
- 미확인 항목

### 5. 남은 리스크
- 아직 확인 못한 부분
- 실제 운영 환경에서 추가 확인이 필요한 부분

## 보고 예시

### 예시 1
- 변경 파일: `UserCreateForm.tsx`
- 검증 항목:
  - 정상 저장 성공
  - 필수값 누락 오류 표시
  - 저장 실패 시 입력값 유지
  - 저장 버튼 연타 방지
  - 모바일 폭 기본 확인
- 테스트 방법:
  - 수동으로 정상/실패/빈 입력 시나리오 실행
- 결과:
  - 위 항목 모두 통과
- 남은 리스크:
  - 실제 운영 API timeout 상황은 mock 기준만 확인

### 예시 2
- 변경 파일: `MatchFilterPanel.tsx`, `MatchListPage.tsx`
- 검증 항목:
  - 검색 결과 없음 empty state
  - 필터 초기화 버튼
  - 목록 로딩 상태
  - 기존 검색 동작 회귀 확인
- 결과:
  - empty/loading/기존 검색 모두 정상
- 남은 리스크:
  - 매우 긴 팀명 데이터는 실운영 데이터로 추가 확인 필요

## 한 줄 기준

좋은 테스트는  
**"코드가 돌아간다"를 확인하는 것이 아니라, "사용자 흐름이 안전하게 유지되고 실패 상황에서도 무너지지 않는지"를 확인하는 것**이다.
