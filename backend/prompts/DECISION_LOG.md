## ADR-001 · AI는 분석 보조, 시스템이 실행 — 권한 경계 확정

- **결정**: AI 모델(Gemini/Opus/GPT)에 매매 실행 권한을 절대 부여하지 않는다.
  AI는 분석·요약·구조화 산출물만 생성하고, 시스템(Python Rule Engine)이 단독으로 주문 권한을 가진다.

- **AI가 할 수 있는 것**:
  - 뉴스/공시/미국장 요약 (Gemini)
  - 시장 톤 판단, 종목 적합도 점수, 복기 해석 (Opus)
  - RulePack JSON 구조화, 자연어 리포트 작성 (GPT)
  - 위 산출물은 모두 디스크의 .md 또는 .json 파일로 저장된다

- **AI가 할 수 없는 것**:
  - KIS API 직접 호출 (주문/조회/체결)
  - 리스크 한도값 변경 (`daily_loss_limit`, `max_positions`, `stop_loss_rate`)
  - 매수/매도/손절/익절 실시간 결정
  - 어제의 RulePack 직접 수정
  - 다른 AI를 직접 호출 (모든 AI 호출은 시스템이 트리거)

- **이유**:
  - AI는 허언/오판/과거 데이터 누락 가능성이 항상 있음
  - 매매 사고는 즉시 자금 손실로 직결되어 복구 불가
  - 검증 가능한 결정론적 코드(룰 엔진)에만 실행 권한을 둬야 사고 추적 가능
  - AI 출력은 항상 "제안서"이고, 시스템은 "승인자 + 실행자"

- **트레이드오프**:
  - AI의 빠른 적응력을 즉시 반영할 수 없음 → 학습/개선은 PM의 수동 검토 후 settings.json 업데이트로만 반영
  - 시스템 검증 로직이 복잡해짐 → 검증 게이트 3단(Schema/Risk/Sanity)으로 분리하여 관리

- **확정일**: 2026-05-01

## ADR-002 · AI 위원회 역할 분담 — Gemini → Opus → GPT 순차 파이프라인

- **결정**: 단일 AI 의존 금지. 역할별로 모델을 분리하여 교차 검증 가능하게 구성한다.

- **역할 매트릭스**:

  | 단계 | 모델 | 입력 | 출력 (파일) |
  |---|---|---|---|
  | 1차: 정보 수집 | Gemini | 뉴스/공시/미국장 raw | `night_brief_*.md`, `news_summary_*.md` |
  | 2차: 정성 판단 | Opus | Gemini 산출물 + 시장 데이터 | `market_tone_*.md`, `screening_*.md`, `review_*.md` |
  | 3차: 구조화 | GPT | Opus 산출물 + 시스템 템플릿 | `rulepack_draft.json`, `daily_report_*.md` |
  | 4차: 검증·실행 | 시스템 | GPT 산출물 | `rulepack_active.json`, 주문 로그 |

- **이유**:
  - 단일 모델은 자기 출력을 검증 못함 → 모델 간 교차 확인이 자연스럽게 됨
  - Gemini는 무료 한도가 있어 장문 요약에 최적, Opus는 정성 판단 강점, GPT는 JSON 구조화 안정성 강점
  - 각 단계가 파일로 분리되어 어느 단계에서 문제 생겼는지 추적 가능

- **트레이드오프**:
  - 처리 시간이 단일 AI 호출보다 길어짐 (대략 2~5분 추가) → 장 시작 1시간 전 08:00부터 시작하므로 영향 없음
  - API 비용이 늘어남 → Gemini 무료 한도 적극 활용으로 상쇄

- **확정일**: 2026-05-01

## ADR-003 · 단계 간 통신 방식 — 파일 기반 트리거

- **결정**: 단계 간 데이터 전달은 디스크 파일을 통해 수행한다. AI가 다른 AI를 직접 호출하는 구조는 금지한다.

- **흐름**:
  ```
  Python Scheduler가 Gemini API 호출
    → 응답 수신 (동기 요청)
    → night_brief_20260501.md 디스크 저장
    → 저장 완료 = 1단계 종료 신호
    → Python Scheduler가 다음 단계(Opus 호출) 시작
  ```

- **이유**:
  - 시스템이 모든 단계의 시작/종료를 손에 쥐고 있어야 함
  - 파일이 곧 감사 추적 자료 (어떤 입력으로 어떤 출력이 나왔는지 영구 보존)
  - 어느 단계에서 실패해도 이전 단계 결과가 디스크에 남아있어 재개 가능
  - 파일이 없으면 = 이전 단계가 실패한 것 → 즉시 감지 가능

- **파일 명명 규칙**:
  - `{단계명}_{YYYYMMDD}_{HHMM}.md` (또는 `.json`)
  - 예: `night_brief_20260501_0800.md`, `rulepack_active_20260501_0850.json`

- **저장 위치**:
  - `data/ai_outputs/YYYY-MM-DD/` 디렉토리 (날짜별 분리)

- **확정일**: 2026-05-01

## ADR-004 · RulePack 검증 3단 게이트

- **결정**: GPT가 생성한 RulePack JSON은 반드시 3단 게이트를 통과해야만 룰 엔진에 적용된다.

- **게이트 1: Schema Validator**
  - 필수 필드 존재 확인
  - 타입 확인 (숫자/문자/배열/boolean)
  - 값 범위 확인 (예: `stop_loss_rate`는 -10% ~ 0% 사이)
  - 실패 시: 즉시 reject, 어제의 RulePack 사용

- **게이트 2: Risk Guard**
  - 하드코딩된 시스템 한도 위반 검사
  - `daily_loss_limit`이 -3%보다 큰가? → 자동 -3%로 덮어씀
  - `max_positions`가 20을 넘는가? → 20으로 덮어씀
  - `stop_loss_rate`가 -1%보다 느슨한가? → -1%로 덮어씀
  - 위반 항목은 경고 로그 기록 (PM이 나중에 확인)

- **게이트 3: Sanity Check**
  - 어제 RulePack과 비교, 변동폭 50% 초과 시 reject
  - 최근 5거래일 데이터로 백테스트 → 손실 -5% 초과 시 reject
  - 실패 시: reject + PM에게 텔레그램 알림

- **이유**:
  - AI가 잘못된 JSON, 비현실적 값, 위험한 룰을 만들어도 사고로 이어지지 않게 차단
  - 게이트 통과 = 시스템이 안전하다고 판정한 것이므로 룰 엔진은 무조건 신뢰하고 실행

- **확정일**: 2026-05-01

## ADR-005 · Risk Circuit Breaker — AI 무관 절대 안전장치

- **결정**: 일일 손실 한도, 최대 포지션 수, 종목별 손절선은 코드에 상수로 박아두고 AI가 절대 변경할 수 없게 한다.

- **하드코딩 상수 (예시값, PM이 최종 결정)**:
  - `DAILY_LOSS_LIMIT = -0.03` (계좌 자산 -3% 도달 시 전 종목 청산 + 신규 매수 차단)
  - `MAX_POSITIONS = 10`
  - `STOP_LOSS_RATE = -0.02` (개별 종목 -2% 도달 시 자동 손절)
  - `MAX_POSITION_SIZE = 0.10` (한 종목당 자산의 최대 10%)

- **변경 절차**:
  - 코드(`config/risk_constants.py`)에서만 변경 가능
  - 변경 시 git commit 필수, PM의 명시적 승인 필요
  - RulePack JSON에 동일 키가 있어도 무시됨 (시스템 상수가 우선)

- **이유**:
  - AI가 "오늘은 공격적으로 가자"고 출력해도 시스템 차원에서 차단되어야 함
  - 매매 사고의 마지막 방어선
  - 변경 이력이 git에 남아 책임 추적 가능

- **확정일**: 2026-05-01

## ADR-006 · AI 학습 루프 — 자동 학습 금지, 수동 반영만 허용

- **결정**: AI 복기 결과를 시스템이 자동으로 RulePack 기본값에 반영하지 않는다.

- **흐름**:
  1. Opus가 복기 리포트 작성 → `review_*.md`
  2. GPT가 패턴을 구조화 → `review_patterns.json`
  3. 시스템이 통계 DB에 누적 (`review_stats.db`)
  4. **주 1회 PM이 직접 검토** → 충분히 쌓인 패턴만 `settings.json` 수동 업데이트

- **금지 사항**:
  - AI가 "내일부터 손절을 -1%로 바꾸세요" 같은 직접 변경 제안을 출력하는 것
  - 시스템이 통계만 보고 자동으로 룰을 변경하는 것
  - 학습 결과를 다음날 RulePack에 자동 반영하는 것

- **이유**:
  - 자동 학습은 과적합/연쇄 오류 위험이 큼 (예: 일시적 시장 이상치를 패턴으로 인식)
  - 룰 변경은 자금 손실로 직결되므로 사람의 판단 필수
  - 통계 누적 자체는 안전, 룰 변경만 사람의 게이트

- **확정일**: 2026-05-01

## ADR-007 · AI 호출 실패 시 폴백 정책

- **결정**: AI 호출이 실패해도 시스템은 멈추지 않는다. 안전한 기본값으로 폴백한다.

- **폴백 우선순위**:
  1. 같은 단계 어제 산출물 캐시 사용
  2. Gemini 실패 시 → Groq 또는 OpenRouter 무료 모델로 대체
  3. 모든 AI 실패 시 → 정량 룰만으로 진행 (시장 톤 = 중립, 유니버스 = 기본 60종목)
  4. RulePack 검증 실패 시 → 어제의 RulePack 그대로 사용
  5. 치명적 실패(KIS 토큰 갱신 실패 등) → 당일 매매 전체 스킵 + PM에게 알림

- **이유**:
  - "AI가 똑똑한가"보다 "제한에 걸려도 시스템이 멈추지 않는가"가 더 중요
  - 자동매매는 운영 시간(09:00~15:30)에 멈추면 안 됨
  - 어제 산출물 = 이미 검증된 데이터이므로 안전한 폴백

- **확정일**: 2026-05-01

## ADR-008 · 프롬프트는 코드에 고정 — 실시간 생성 금지

- **결정**: 시스템이 AI를 호출할 때 사용하는 프롬프트는 모두 `prompts/` 디렉토리에 사전 정의된 템플릿 파일을 사용한다. 코드 실행 중 동적으로 프롬프트를 조립하지 않는다.

- **구조**:
  ```
  prompts/
  ├── 0800_gemini_news_summary.md
  ├── 0805_opus_market_tone.md
  ├── 0830_opus_screening.md
  ├── 0845_gpt_rulepack_generation.md
  ├── 1600_opus_review.md
  ├── 1630_gpt_daily_report.md
  ├── 2200_gemini_us_market_brief.md
  └── fallback/
      ├── groq_news_summary.md
      └── openrouter_backup.md
  ```

- **이유**:
  - 프롬프트가 곧 시스템과 AI 사이의 계약서
  - 출력 포맷이 고정되어야 검증 게이트(JSON Schema) 설계 가능
  - 프롬프트 변경 이력이 git에 남아 추적 가능
  - "어떤 프롬프트로 어떤 결과가 나왔나"를 항상 재현 가능

- **변경 절차**:
  - 프롬프트 변경은 코드 변경과 동일 취급 (git commit + PM 승인)
  - 변경 후 최소 3일 백테스트 후 운영 적용

- **확정일**: 2026-05-01
