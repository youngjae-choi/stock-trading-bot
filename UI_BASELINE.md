# UI_BASELINE.md

프로젝트 UI 작업의 단일 기준 문서.
목적은 일관성 유지, 현장 사용성 확보, 회귀 버그 방지다.

## 1. 적용 범위
- 개별 기능 명세가 없으면 본 문서를 기본값으로 적용한다.
- 개별 명세가 있을 경우, 개별 명세를 우선하고 본 문서는 보완 기준으로 적용한다.

## 2. 핵심 원칙
- 기능 동작 + 사용성 + 오류 복구 가능성까지 포함해야 완료다.
- 사용자 상태 피드백 없는 침묵 UI는 금지한다.
- 예외 처리는 특별 케이스가 아니라 기본 흐름으로 설계한다.
- 사용자 메시지와 개발자 로그를 분리한다.
- 같은 성격의 UI는 같은 패턴을 따른다.

## 3. 공통 상태 규약
- Initial: 첫 진입 시 사용자 행동이 명확해야 한다.
- Loading: 진행 상태가 보여야 하고, 중복 요청을 막아야 한다.
- Empty: 에러와 구분된 빈 상태 + 다음 행동 CTA를 제공한다.
- Error: 실패 원인 범주를 구분하고 재시도 수단을 제공한다.
- Success: 성공 사실과 UI 반영 결과가 즉시 보이도록 한다.

## 4. 버튼/폼 규약
- 버튼 라벨은 동사 중심으로 명확하게 작성한다.
- 처리 중 중복 클릭을 막는다.
- 비활성 버튼은 이유를 사용자에게 전달한다(tooltip/문구).
- 아이콘 버튼은 `aria-label`을 필수로 둔다.
- 폼은 필수값/형식/범위/중복 검증을 지원한다.
- 제출 실패 시 입력값은 가능한 유지한다.

## 6. 레이아웃/반응형 기준
- 전체 레이아웃 최소 폭 기준을 정한다.
- 붕괴형 반응형보다 구조 유지 + 스크롤 처리를 우선한다.

## 7. 접근성 기준
- 주요 인터랙션 요소는 키보드 접근 가능해야 한다.
- 포커스 상태가 시각적으로 보여야 한다.
- 색상만으로 상태를 전달하지 않는다.
- 에러 메시지는 위치와 의미가 명확해야 한다.

## 8. 로그/관측 기준
- 로그는 시작/성공/실패를 분리해 남긴다.
- 에러 로그 예: `console.error('[ERROR]', location, '-', error.message)`
- 비밀번호/토큰/개인정보 raw 로그는 금지한다.

## 9. 테스트 게이트 (UI 변경 시 필수)
1. 빌드 통과
2. E2E 테스트 통과
3. 수동 점검 통과 (프로젝트별 핵심 시나리오)

## 10. 작업 운영 규칙
- BUG는 순차 처리하고 BUG별 PASS 후 다음으로 진행한다.
- 실패 시 수정 -> 재테스트를 PASS할 때까지 반복한다.
- 변경 결과는 `docs/agent-comm/OUTBOX_Oracle.md`에 BUG별 기록한다.

## 11. 구현 보고 최소 항목

- 예외 처리와 롤백/복구 전략
- 접근성 대응 항목
- 테스트 결과 및 잔여 리스크
