# FEATURE_TEMPLATE.md

## 목적

이 문서는 새로운 기능 개발, 기존 기능 수정, 버그 수정 요청 시  
AI 에이전트가 **충분한 맥락**, **명확한 완료 기준**, **공통 품질 기준**을 가지고 작업하도록 하기 위한 템플릿이다.

목표는 다음과 같다.

- 기능 요청을 막연한 지시가 아니라 실행 가능한 작업 단위로 바꾼다
- 요구사항 누락으로 인한 품질 저하를 줄인다
- UI, 예외 처리, 테스트, 완료 기준을 함께 정의한다
- AI 에이전트가 "돌아가는 코드"만 만들고 끝내지 않게 한다

이 문서는 기능 요청 시 복사해서 작성하는 기본 템플릿이다.  
필요 없는 항목은 삭제하거나 "없음"으로 표시할 수 있다.  
하지만 **핵심 목표 / 완료 기준 / 예외 처리 / 테스트 방법**은 가능한 한 유지한다.

# 기능 요청 템플릿

## 1. 작업명

예:
- 회원 생성 폼 구현
- 경기 필터 UI 추가
- Telegram 전송 실패 재시도 기능 추가
- 리스트 검색 성능 개선
- 삭제 확인 모달 수정

---

## 2. 작업 유형

하나를 선택한다.

- 신규 기능
- 기존 기능 개선
- UI 개선
- 버그 수정
- 리팩토링
- 성능 개선
- 안정화 / 예외 처리 강화
- 테스트 보강

---

## 3. 배경 / 왜 필요한가

이 기능이 왜 필요한지 적는다.  
단순히 "만들어줘"보다 이유가 있어야 AI가 더 나은 판단을 할 수 있다.

예:
- 사용자가 저장 후 성공 여부를 알기 어렵다
- 현재 목록에서 검색 결과가 없을 때 빈 화면만 나온다
- Telegram 전송 실패 시 재시도 방법이 없다
- 모바일에서 필터 UI가 깨진다
- 사용자가 같은 버튼을 여러 번 눌러 중복 요청이 발생한다

---

## 4. 핵심 목표

이 작업의 진짜 목적을 한두 문장으로 적는다.

예:
- 사용자가 저장 과정과 결과를 명확히 인지할 수 있게 한다
- 경기 리스트에서 조건 검색과 빈 상태 처리를 자연스럽게 만든다
- 실패하더라도 사용자가 다시 시도할 수 있게 한다

---

## 5. 범위

### 이번 작업에 포함
이번 작업에서 반드시 할 것

예:
- 저장 버튼 로딩 처리
- 중복 클릭 방지
- 성공/실패 토스트
- 폼 검증 메시지
- API 실패 시 에러 UI

### 이번 작업에 제외
이번 작업에서는 하지 않을 것

예:
- 서버 API 구조 변경
- 디자인 시스템 전면 개편
- 권한 체계 재설계
- 다국어 지원

## 7. 현재 문제점

현재 어떤 문제가 있는지 구체적으로 적는다.

예:
- 저장 버튼 클릭 후 반응이 없어 보인다
- 실패 시 입력값이 모두 사라진다
- 검색 결과가 없을 때 표만 비어 있다
- API timeout 시 사용자가 원인을 알 수 없다

## 8. 기대 동작

기능이 완료되면 사용자가 어떻게 느껴야 하는지, 어떤 흐름이 되어야 하는지 적는다.

예:
- 사용자가 저장 버튼을 누르면 즉시 로딩 상태를 본다
- 저장 실패 시 어떤 문제인지 안내받고 다시 시도할 수 있다
- 검색 결과가 없으면 empty state와 다음 행동 버튼이 보인다
- 삭제 전 확인 모달이 뜬다
- 모바일에서도 핵심 기능 수행이 가능하다

## 9. 기능 요구사항

가능한 한 구체적으로 적는다.

예:

- 요청 중 버튼 문구를 "저장 중..."으로 바꾼다
- 성공 시 토스트를 띄우고 목록 페이지로 이동한다
- 실패 시 폼 값을 유지한다
- 필수값 누락 시 각 필드 아래에 에러 메시지를 표시한다
- 검색 결과가 없으면 empty state를 표시한다
- 삭제 버튼 클릭 시 확인 모달을 띄운다

## 10. UI 요구사항

## 21. 오케스트레이션 지시 항목 (프로젝트 특화)

작업 요청 시 아래를 함께 적으면 실행 품질이 크게 올라간다.

- 처리 방식:
  - 순차 처리(기본) / 병렬 처리(명시 시)
- BUG 처리 규칙:
  - `BUG-01 -> BUG-02 -> ...` 순차
  - BUG별 테스트 PASS 전 다음 단계 금지
- 역할 분리:
  - Explore(분석) -> Executor(구현) -> Oracle(검증)
- 결과 기록 위치:
  - `OUTBOX_[페르소나].md` 최상단
- 종료 조건:
  - build/e2e/수동 검증 전부 PASS

실행 프롬프트 예시:
```bash
codex exec --dangerously-bypass-approvals-and-sandbox "docs/agent-comm/INBOX_[페르소나].md 읽고 최신 TASK를 순차 수행해. BUG별 테스트 실패 시 수정/재테스트를 반복하고, 최종 build/e2e/수동 검증까지 통과한 뒤 결과를 docs/agent-comm/OUTBOX_[페르소나].md 최상단에 기록해."
```

# 작성 예시 1

## 작업명
회원 생성 폼 저장 UX 개선

## 작업 유형
기존 기능 개선

## 배경 / 왜 필요한가
현재 저장 버튼을 눌러도 반응이 느려 보이고 실패 시 입력값이 사라져 사용자 경험이 좋지 않다.

## 핵심 목표
사용자가 저장 과정과 결과를 명확히 인지하고, 실패해도 다시 시도할 수 있게 만든다.

## 이번 작업에 포함
- 저장 버튼 로딩 처리
- 중복 클릭 방지
- 필드 검증 메시지
- 저장 실패 시 입력값 유지
- 성공/실패 토스트

## 이번 작업에 제외
- 서버 API 변경
- 회원 목록 페이지 리디자인

## 현재 문제점
- 저장 중 상태 표시 없음
- 실패 시 폼 초기화
- 필수값 누락 메시지 부재

## 기대 동작
- 저장 버튼 클릭 시 즉시 로딩 상태가 보인다
- 저장 실패 시 입력값이 유지된다
- 필수값 누락 시 필드별 메시지가 보인다
- 성공 시 토스트 후 목록으로 이동한다

## 완료 기준
- 저장 버튼 클릭 시 1회만 요청된다
- 저장 중 버튼은 비활성화된다
- 필수값 누락 시 각 필드 아래 오류 메시지가 표시된다
- 저장 실패 시 입력값이 유지된다
- 성공 시 토스트가 보이고 목록 페이지로 이동한다

## 테스트 방법
- 필수값 없이 저장
- 정상 저장
- 실패 응답 mock 후 저장
- 버튼 연속 클릭
- 모바일 폭 확인

## 이번 작업에 제외
- 검색 알고리즘 변경
- 필터 조건 추가

## 기대 동작
- 검색 결과가 없으면 안내 문구가 보인다
- 필터 초기화 버튼이 보인다
- 초기 데이터가 없는 상태와 검색 결과 없음 상태를 구분한다

## 완료 기준
- 결과 없음 상태에서 empty state가 보인다
- 필터 초기화 버튼이 동작한다
- 초기 빈 상태와 검색 결과 없음 상태가 다른 문구로 표시된다

## 테스트 방법
- 전체 데이터 0건
- 검색 결과 0건
- 필터 초기화 동작 확인
- 모바일 화면 확인

# AI 에이전트용 사용 지침

AI 에이전트는 기능 요청을 받으면 다음 순서로 작업한다.

1. 작업의 핵심 목표를 먼저 이해한다
2. 범위와 제외 범위를 구분한다
3. UI_BASELINE.md와 ERROR_HANDLING.md 기준을 기본 적용한다
4. loading / empty / success / error 상태를 점검한다
5. 예외 처리와 완료 기준이 충족되는지 확인한다
6. 구현 후 보고 형식에 맞게 정리한다
7. 가능하면 테스트 또는 최소 수동 검증 방법을 함께 제시한다
