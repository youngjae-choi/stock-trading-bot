## 필수 작업 프로세스 (반드시 이 순서를 지킨다)

모든 개발 및 수정은 아래 순서를 반드시 따른다.
순서를 건너뛰거나 임의로 변경하는 것은 금지한다.
```
1. 개발/수정 발생
        ↓
2. 새 작업 시작 전 커밋 여부 확인 ← Claude가 PM에게 반드시 질문
   "현재 상태를 커밋(저장)하시겠습니까?"
   PM이 "세이브해줘" → Claude가 git commit 실행 후 진행
   PM이 "그냥 진행해" → 커밋 없이 진행
   ※ git commit 실행 권한은 Claude(Sisyphus)만 가진다
      Gemini / Codex는 git commit 금지
        ↓
3. 기획서 업데이트 (PM이 작성)
        ↓
4. 작업계획서 + 테스트계획서 작성 (Prometheus가 작성, PM이 승인)
        ↓
5. 개발 (Executor / Frontend가 구현)
        ↓
6. 테스트 (Oracle이 코드 리뷰, PM이 브라우저 동작 확인)
        ↓
7. 이상 없으면 → 테스트결과서 작성 (Prometheus가 작성)
   이상 있으면 → 5번(개발)으로 돌아가 반복
        ↓
8. 종료 → 매뉴얼 + 기획서 + WBS 업데이트
```

## 개발 주의사항 (반드시 준수)

### 1. 주석 작성 규칙
- 모든 함수/컴포넌트에 목적과 파라미터를 설명하는 주석을 작성한다
- 비즈니스 로직이 복잡한 경우 처리 흐름을 단계별로 주석으로 설명한다
- 임시 코드나 workaround에는 반드시 이유와 TODO를 주석으로 남긴다
- 주석 없는 코드는 미완성으로 간주한다

### 2. 단계별 로그 규칙
- 모든 주요 작업은 시작/완료/실패를 서버 로그로 기록한다
- 문제 발생 시 로그만으로 원인을 파악할 수 있어야 한다
- 로그 레벨을 구분한다: INFO / WARN / ERROR
- 클라이언트 오류는 서버 API Route에서 반드시 로깅한다

### 3. 상상 코드 금지
- 실제로 확인하지 않은 API, 라이브러리 메서드, 설정값을 임의로 작성하지 않는다
- 공식 문서에서 확인한 내용만 구현한다
- "아마 이렇게 동작할 것이다"는 구현 근거가 되지 않는다

### 4. 허위 판단 금지
- 동작 확인 없이 "완료됐습니다"라고 보고하지 않는다
- 테스트 결과를 추측으로 작성하지 않는다
- 불확실한 부분은 반드시 "확인 필요"로 명시한다

### 5. 기존 기능 파괴 금지
- 새 기능 추가 시 기존 기능의 동작을 반드시 확인한다
- 파일 수정 전 해당 파일이 다른 곳에서 어떻게 사용되는지 먼저 파악한다
- 기존 API Route, 컴포넌트 인터페이스, DB 스키마는 하위 호환성을 유지한다

**Oracle** (@oracle)

## PM과의 협업 규칙

1. **작업 전 계획 보고**: 코드 작성 전 반드시 작업계획서를 먼저 보여주고 PM 승인을 받는다
2. **모호하면 질문**: 추측으로 진행하지 않는다
3. **완료 후 다음 추천**: 작업 완료 후 "다음 추천 작업" 3가지를 제시한다
4. **에러 보고 순서**: 서버 로그 확인 → 원인 설명 → 해결책 제안
5. **기술 용어 설명**: PM이 비개발자임을 인지하고 기술 결정을 쉬운 말로 설명한다

## 의사결정 게이트

### Gate 1: 해석 게이트
- PM의 요청을 올바르게 이해했는가
- 모호한 부분이 있는가
- 모호하면 구현하지 말고 질문한다

### Gate 2: 범위 게이트
- 작업계획서와 테스트계획서가 PM 승인을 받았는가
- 기술 스택 제약을 지키는가
- 승인 전에 구현하지 않는다

### Gate 3: 진실 게이트
- 실제 활성 코드 경로를 파악했는가
- 공식 문서에서 API를 확인했는가
- 파악 전에 수정하지 않는다

### Gate 4: 위험 게이트
- 보안 규칙을 위반하지 않는가
- 기존 기능이 깨지지 않는가
- 위험하면 PM에게 먼저 보고한다

### Gate 5: 검증 게이트
- 실제로 동작하는지 확인했는가
- 서버 로그에 정상 흐름이 찍히는가
- "코드를 작성했다"는 완료가 아니다

### Gate 6: 인계 게이트
- 테스트결과서가 작성되었는가
- 문서가 업데이트되었는가
- PM이 무엇이 바뀌었는지 이해할 수 있는가

## 완료의 정의

작업은 아래 모든 항목이 충족될 때만 완료다:

- 작업계획서와 테스트계획서가 PM 승인을 받았다
- 공식 문서 기반으로 구현했다 (상상 코드 없음)
- 모든 함수에 주석이 작성되었다
- 단계별 로그가 서버에서 확인 가능하다
- 기존 기능이 깨지지 않았다
- 테스트계획서의 모든 시나리오를 통과했다
- 테스트결과서가 작성되었다
- PM이 결과를 확인하고 승인했다

코드 작성 ≠ 완료
동작 확인 + 로그 확인 + 문서 업데이트 + PM 승인 = 완료
