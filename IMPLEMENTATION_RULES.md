# IMPLEMENTATION_RULES.md

## 목적

이 문서는 기능 추가, 버그 수정, 리팩토링, 구조 개선 작업 시  
AI 에이전트와 개발자가 따라야 하는 **구현 방식의 공통 규칙**을 정의한다.

목표는 다음과 같다.

- AI가 임의로 구조를 바꾸거나 과도하게 확장하는 것을 방지한다
- 기존 코드베이스와 작업 맥락을 존중하도록 강제한다
- 수정 방식, 파일 처리 방식, 주석, 로그, 테스트, 보고 형식을 일관되게 유지한다
- "그럴듯한 새 구조"보다 "현재 프로젝트에 맞는 안전한 구현"을 우선하게 만든다
- 구현 품질과 유지보수성을 함께 확보한다

이 문서는 개별 기능 명세보다 우선하지 않는다.  
다만 별도 지시가 없으면 본 문서 기준을 기본값으로 따른다.

---

## 핵심 원칙

### 1. 기존 구조를 먼저 이해하고, 그 다음에 수정한다

AI 에이전트는 먼저 현재 구조를 이해해야 한다.  
이해 없이 새 구조를 제안하거나 구현하면 안 된다.

반드시 먼저 확인할 것:
- 현재 진입점
- 데이터 흐름
- 책임 분리 방식
- 기존 네이밍 규칙
- 공용 함수/공용 컴포넌트 존재 여부
- 비슷한 기능의 기존 구현 방식
- 이미 존재하는 예외 처리 방식
- 기존 테스트 구조

금지:
- 기존 구조를 읽지 않고 새 디렉터리/새 패턴 도입
- 이미 있는 공통 로직을 무시하고 중복 구현
- 현재 프로젝트 문맥과 무관한 "이상적인 구조" 강요

---

### 2. 최소 변경으로 목적을 달성하는 것을 기본값으로 한다

좋은 구현은 무조건 크게 바꾸는 것이 아니다.

기본 원칙:
- 이번 작업 범위 안에서 해결 가능한 최소 변경을 우선한다
- 영향 범위가 넓은 리팩토링은 명시적 요청이 있을 때만 한다
- 구조 개선이 필요하더라도, 기능 요구와 직접 관련 없는 대규모 개편은 피한다

예:
- 버튼 로딩 처리 추가가 목적이면 상태관리 전체 재구성 금지
- 에러 메시지 개선이 목적이면 API 추상화 전면 개편 금지
- collector 보강이 목적이면 파이프라인 전체 재설계 금지

---

### 3. 추측하지 말고, 근거 있는 구현만 한다

AI 에이전트는 없는 파일, 없는 함수, 없는 API, 없는 데이터 구조를 상상해서 구현하면 안 된다.

반드시 지켜야 할 것:
- 실제 존재하는 코드와 파일 기준으로 구현
- 현재 프로젝트에서 확인 가능한 이름을 사용
- 확인되지 않은 값은 방어적으로 처리
- 불확실한 내용은 가정임을 명시

금지:
- 없는 helper를 있는 것처럼 사용
- 없는 endpoint를 임의로 호출
- 기존 코드와 무관한 새로운 서비스 계층 발명
- 실제 확인 없이 "아마 이럴 것" 기반 구현

---

### 4. 구현은 기능 요구사항 + UI/오류/테스트 규약을 함께 만족해야 한다

기능 구현은 단순히 요구사항 몇 줄만 맞추면 끝이 아니다.

기본적으로 함께 따라야 하는 문서:
- `AGENTS.md`
- `UI_BASELINE.md`
- `ERROR_HANDLING.md`
- `FEATURE_TEMPLATE.md`
- `TEST_RULES.md`

즉 구현 완료의 의미는:
- 기능이 동작함
- 기본 UI 상태가 처리됨
- 오류 처리가 있음
- 테스트 또는 검증이 수행됨
- 보고가 가능함

---

### 5. 완료보다 안전한 완료를 우선한다

겉으로는 빨리 끝난 것처럼 보여도,
- 기존 기능을 깨뜨리거나
- 예외 처리가 없거나
- 로그가 없거나
- 테스트가 없으면

좋은 구현이 아니다.

원칙:
- 속도보다 안정성
- 화려함보다 유지보수성
- 새로움보다 일관성
- 과한 일반화보다 명확한 해결

---

## 작업 시작 전 필수 절차

AI 에이전트는 구현 전에 최소 아래를 수행한다.

### 1. 작업 목표 파악
- 무엇을 바꾸는 작업인가
- 왜 필요한가
- 완료 기준은 무엇인가
- 이번 작업 범위와 제외 범위는 무엇인가

### 2. 관련 코드 확인
- 영향을 받는 파일 탐색
- 진입점 확인
- 비슷한 구현 사례 확인
- 공통 유틸/공통 컴포넌트 존재 여부 확인

### 3. 영향 범위 판단
- 직접 영향 파일
- 간접 영향 파일
- 공용 로직 영향 여부
- 테스트 영향 여부

### 4. 구현 전략 선택
- 부분 수정으로 해결 가능한가
- 파일 전체 교체가 더 안전한가
- 구조 변경이 꼭 필요한가
- 리스크가 어디에 있는가

---

## 파일 수정 원칙

### 1. 기본은 기존 파일을 존중한 수정

가능하면 기존 파일의 역할을 유지하면서 수정한다.

원칙:
- 기존 책임을 크게 벗어나지 않음
- 이미 있는 유틸은 재사용
- 비슷한 패턴은 기존 코드와 맞춤
- 같은 레이어의 책임을 다른 레이어로 무분별하게 옮기지 않음

---

### 2. 완전교체 vs 부분수정 기준

사용자가 **완전교체**를 선호하거나, 아래 조건이면 완전교체를 우선 검토한다.

완전교체가 더 적합한 경우:
- 기존 파일 구조가 크게 꼬여 있음
- 수정 범위가 파일 전반에 걸침
- 부분 패치가 오히려 실수 위험이 큼
- 비개발자가 복붙 적용해야 함
- 사용자가 명시적으로 "완전 교체본"을 요청함

부분수정이 더 적합한 경우:
- 수정 범위가 매우 좁음
- 파일이 크고 나머지 부분은 안정적임
- 전체 교체 시 충돌 위험이 큼
- 현재 코드 문맥 유지가 더 중요함

원칙:
- 어느 방식이 더 안전한지 판단해서 선택
- 선택 이유를 짧게 설명 가능해야 함

---

### 3. 새 파일 생성 기준

새 파일은 꼭 필요할 때만 만든다.

새 파일 생성 허용 조건:
- 책임 분리가 명확히 좋아짐
- 기존 파일이 과도하게 비대함
- 재사용 가능한 독립 단위가 분명함
- 프로젝트 구조와 일관됨
- 명시적 요청이 있음

금지:
- 사소한 이유로 파일 쪼개기
- helper 남발
- util, common, manager, service 같은 모호한 이름의 파일 무분별 생성
- 사용처가 불명확한 추상화 파일 생성

---

### 4. 파일명 / 함수명 / 변수명 규칙

원칙:
- 현재 프로젝트 네이밍 스타일을 따른다
- 역할이 드러나는 이름을 쓴다
- 지나치게 짧거나 추상적인 이름을 피한다
- 한 파일 안에서 동일 개념은 동일 용어를 사용한다

좋은 예:
- `collect_market_v1`
- `build_error_message`
- `is_retryable_error`
- `userCreateFormState`

나쁜 예:
- `doThing`
- `handleStuff`
- `tempData`
- `managerUtil`
- `helper2`

---

## 구조 변경 원칙

### 1. 구조 변경은 기본값이 아니다

사용자가 구조 개선을 명시적으로 요청하지 않았다면,  
기능 구현을 위해 꼭 필요한 수준 이상으로 구조를 바꾸지 않는다.

금지:
- 폴더 구조 전면 변경
- 대규모 이름 변경
- 공통 추상화 남발
- 레이어 재설계
- 프레임워크 스타일 변경

---

### 2. 구조 변경이 필요한 경우

다음 조건이 동시에 강하면 구조 변경을 검토할 수 있다.

- 현재 구조로는 반복 버그가 발생함
- 이번 작업을 넣으면 코드가 더 위험해짐
- 중복이 과도함
- 책임 분리가 명확히 무너져 있음
- 변경 이득이 리스크보다 큼

이 경우에도:
- 한 번에 전면 개편하지 않음
- 현재 작업과 직접 관련된 범위 안에서만 조정
- 영향 범위를 설명 가능해야 함

---

## 주석 규칙

### 1. 주석은 "무엇을 하는가"보다 "왜 이렇게 하는가"에 가깝게 쓴다

좋은 주석은 이유와 판단 근거를 남긴다.

좋은 예:
- 외부 API가 간헐적으로 null을 반환하므로 방어 처리
- 중복 제출을 막기 위해 요청 중 버튼 비활성화
- 기존 화면 흐름 유지 위해 성공 후 페이지 이동 대신 상태만 갱신

나쁜 예:
- 변수에 값 할당
- 함수 호출
- 리스트 순회

---

### 2. 주석이 필요한 경우

- 복잡한 분기
- 비직관적 예외 처리
- 외부 시스템 제약 회피
- 임시 우회 로직
- 운영 이슈 반영 코드
- 성능/안정성 목적의 특별 처리

---

### 3. 금지되는 주석

- 코드가 이미 말해주는 수준의 주석
- 오래되면 거짓말이 되기 쉬운 주석
- TODO만 남기고 설명 없음
- 임시 처리인데 이유 없는 주석

---

## 로그 규칙

### 1. 로그는 디버깅 가능한 정보여야 한다

로그는 문제 추적에 도움이 되어야 한다.

권장:
- 기능명
- 액션명
- 상태
- 실패 단계
- 요청 식별자
- 재시도 여부
- 결과 요약

예:
- `[user-create] submit start`
- `[telegram-send] retrying after timeout`
- `[collector-router] unsupported league skipped`

---

### 2. 로그 레벨을 구분한다

가능하면 아래 성격을 구분한다.

- info: 정상 흐름 주요 상태
- warn: 복구 가능 이상 상태
- error: 실패 또는 치명적 문제
- debug: 필요 시 상세 추적

---

### 3. 로그 금지사항

- 민감정보 raw 출력
- 의미 없는 로그 남발
- 같은 오류를 과도하게 중복 출력
- 성공처럼 보이게 하는 잘못된 로그
- catch 후 로그 없이 종료

---

## 오류 처리 구현 원칙

오류 처리는 `ERROR_HANDLING.md`를 따르되, 구현 관점에서 아래를 추가로 지킨다.

### 1. 방어적으로 코딩한다
- null/undefined 확인
- 응답 필드 누락 대비
- 외부 API 변동 대비
- 사용자 입력 이상치 대비

### 2. 실패를 삼키지 않는다
- 적절한 사용자 메시지 또는 로그를 남김
- 필요한 경우 상위 호출부로 전달
- 무조건 silent fail 금지

### 3. 부분 성공을 구분한다
- 생성 성공 + 후속 처리 실패
- 저장 성공 + 알림 실패
- 기본 데이터 성공 + 부가 데이터 실패

이 경우 전체 성공처럼 처리하지 않는다.

---

## UI 구현 원칙

UI는 `UI_BASELINE.md`를 따르되, 구현 관점에서 아래를 추가로 지킨다.

### 1. 상태를 기준으로 구현한다
기능 단위마다 최소 아래 상태를 검토한다.
- initial
- loading
- empty
- success
- error

### 2. 침묵하는 버튼 금지
- 클릭 시 즉시 반응이 보여야 함
- 처리 중 상태 표현
- 중복 클릭 방지

### 3. 실패 후 복구 가능성 고려
- 재시도 버튼
- 입력값 유지
- 이전 상태 유지 여부 검토

---

## 테스트 구현 원칙

테스트는 `TEST_RULES.md`를 따르되, 구현 시 아래를 추가로 지킨다.

### 1. 구현 후 최소 검증은 필수
- 정상 케이스
- 실패 케이스
- 경계 케이스
- 회귀 위험 영역

### 2. 자동 테스트 가능 여부를 먼저 판단
특히 아래는 자동 테스트 우선 검토:
- 순수 함수
- 검증 로직
- 상태 분기
- 공용 유틸
- 회귀 가능성 높은 버그 수정

### 3. 테스트하지 못한 것은 숨기지 않는다
- 미검증 항목 명시
- 이유 명시
- 운영 확인 필요 여부 명시

---

## 리팩토링 원칙

### 1. 리팩토링은 목적이 분명해야 한다

허용:
- 가독성 개선
- 중복 제거
- 책임 분리
- 테스트 가능성 향상
- 오류 처리 명확화

금지:
- 취향 기반 코드 스타일 변경
- 필요 없는 추상화 추가
- 이번 작업과 무관한 정리
- 파일 이동만 많은 리팩토링

---

### 2. 리팩토링 시 기능 동일성 유지

반드시 확인:
- 기존 기능 유지
- 기존 예외 처리 유지 또는 개선
- 로그/보고 흐름 유지
- 테스트 기준 유지

---

## 의존성 추가 원칙

새 라이브러리, 패키지, 프레임워크 추가는 신중해야 한다.

허용 조건:
- 기존 수단으로 해결하기 어렵다
- 유지보수 이득이 분명하다
- 프로젝트와 잘 맞는다
- 번들/배포/운영 리스크를 감수할 가치가 있다
- 사용자가 허용했거나 프로젝트 정책에 맞다

금지:
- 간단한 유틸 때문에 무거운 라이브러리 추가
- 검증 안 된 패키지 도입
- 중복 기능 라이브러리 추가
- "편해서"만으로 의존성 추가

## 성능 고려 원칙

성능 최적화는 필요할 때만 한다.  
다만 명백한 비효율은 방치하지 않는다.

기본 원칙:
- 과한 조기 최적화 금지
- 반복 렌더링, 중복 호출, 불필요한 재계산은 검토
- 네트워크 요청 중복 방지
- 큰 데이터 처리 시 비용 인지
- 외부 API 호출 횟수 절제

## 보안 / 민감정보 원칙

### 1. 민감정보를 직접 노출하지 않는다
- 비밀번호
- 토큰
- API key
- 개인정보
- 결제 정보
- 내부 관리자 식별 정보

### 2. 로그/에러/화면 어디에도 raw 노출 금지

### 3. 사용자 권한을 가정하지 않는다
- 클라이언트 단 검증만 믿지 않음
- 권한 분기 시 방어적으로 처리
- 숨김 처리만으로 보안 완료라고 보지 않음

## 구현 보고 원칙

AI 에이전트는 구현 완료 후 최소 아래를 보고한다.

### 1. 변경 파일
- 수정 파일 목록
- 새로 만든 파일 목록
- 삭제 파일 목록

### 2. 구현 내용
- 어떤 기능을 바꿨는지
- 어떤 흐름을 개선했는지
- 어떤 상태를 처리했는지

### 3. 예외 처리
- 어떤 실패/예외를 처리했는지
- 재시도/롤백 여부
- 부분 성공 처리 여부

### 4. 테스트
- 수동 테스트 내용
- 자동 테스트 여부
- 확인 못한 항목

### 5. 남은 리스크
- 운영 데이터에서 추가 확인이 필요한 부분
- 외부 API 의존 리스크
- 추후 개선 후보

---

## 구현 금지사항

다음은 특별한 이유가 없는 한 금지한다.

- 없는 파일/함수/API를 상상해서 구현
- 기존 구조 무시한 대규모 개편
- 과도한 추상화
- helper/util 남발
- 테스트 없이 완료 처리
- 로그 없이 실패 삼키기
- 부분 성공을 전체 성공처럼 처리
- 민감정보 출력
- 사용자가 복붙해야 하는데 파편적 수정만 제공
- 이유 없는 새 의존성 추가
- 기능 요구와 무관한 리팩토링 확장
- 한 번에 너무 많은 영역 건드리기

## AI 에이전트 작업 지침

AI 에이전트는 구현 시 다음을 따른다.

1. 먼저 현재 구조를 이해한다
2. 실제 존재하는 코드와 파일을 기준으로 작업한다
3. 최소 변경으로 목적 달성을 우선한다
4. 사용자가 완전교체를 선호하면 전체 교체본을 우선 고려한다
5. UI, 오류 처리, 테스트 규약을 함께 적용한다
6. 추측보다 근거 있는 구현을 한다
7. 구현 후 무엇을 바꿨는지와 무엇을 검증했는지 분명히 보고한다

---

## 구현 전 체크리스트

- 작업 목표가 분명한가
- 범위와 제외 범위를 이해했는가
- 관련 파일을 확인했는가
- 비슷한 기존 패턴을 확인했는가
- 최소 변경으로 해결 가능한가
- 완전교체/부분수정 중 더 안전한 방식을 골랐는가

---

## 구현 후 체크리스트

- 기능 요구사항을 충족하는가
- UI 상태를 처리했는가
- 오류 처리를 넣었는가
- 로그가 필요한 곳에 있는가
- 민감정보 노출 위험이 없는가
- 테스트 또는 수동 검증을 했는가
- 기존 기능 영향 범위를 확인했는가
- 보고 가능한 형태로 정리했는가
